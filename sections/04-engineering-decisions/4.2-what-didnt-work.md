## ❀ Failure Analysis Method ────୨ৎ────────୨ৎ────

During the development of Cheese, several ideas looked useful in theory but became unreliable once the robot was tested on the track. Because of this, we learned that a failed test is only valuable if it helps explain what needs to change.

Instead of changing random parts after every crash, we tried to connect each failure to a specific cause. We asked what failed, why it failed, what risk it created, what we changed, and what the failure taught us. This helped us avoid guessing and made the design process more intentional.

The purpose of this section is not to show that Cheese was perfect from the beginning. It is to show how each failed idea influenced the final version of the robot. Many of the strongest decisions in Cheese v3 came directly from things that did not work in v1, v2, or early v3 testing.

---

## ❀ Failure Overview ────୨ৎ────────୨ৎ────

The main failures were connected to four areas: control architecture, software structure, sensing reliability, and mechanical stability. Some problems were caused by hardware limitations, while others came from code complexity, lighting conditions, or movement behavior.

<div align="center">

| What Did Not Work | Main Cause | Final Response |
| :--- | :--- | :--- |
| **Arduino as the full main controller** | Materials arrived late and full integration became risky. | Keep EV3 as the main controller and use Arduino Nano only as a vision bridge. |
| **Very long code versions** | Too many conditions created behavior conflicts. | Reduce the logic from about 1500 lines to about 300 lines. |
| **Infrared front sensor in v1** | Ambient light disrupted readings. | Stop depending on infrared for front detection. |
| **Front ultrasonic sensor in v2** | It detected too many changes during curves. | Remove front ultrasonic dependency and use color-based curve timing. |
| **HuskyLens without lighting** | Obstacle colors appeared too dark or distorted. | Add lighting support for camera visibility. |
| **One light position** | One lamp could not support both camera and floor detection. | Use a dual-light support system. |
| **Weak pin connections** | Pins broke from pressure, force, or repeated stress. | Reinforce loaded joints and treat pins as mechanical failure points. |
| **Wide curves** | Curve behavior was not tight or fluid enough. | Tune steering behavior and post-curve recovery. |
| **Parking failure** | Final stop logic did not place the robot correctly. | Treat parking as its own system that needs testing. |

</div>

These failures are connected because each one showed us that Cheese needed more than individual fixes. The robot needed a clearer system where hardware, software, sensors, structure, and movement logic worked together.

---

## ❀ 1. Arduino as the Full Main Controller ────୨ৎ────────୨ৎ────

At the beginning of the project, we wanted to use Arduino as the full main controller for Cheese. This idea seemed strong because Arduino offers more flexibility with electronics, external modules, and communication. It could have allowed us to build a more custom control system.

However, the problem was timing and reliability. Some materials arrived late, and a full Arduino-based robot would require a lot of extra integration. We would have needed to connect and test motors, sensors, camera communication, power distribution, and code behavior under one new control system. Since competition preparation already required many mechanical and software changes, this became too risky.

The main issue was not that Arduino was a bad platform. The issue was that we did not have enough time to make it dependable as the full control system. A controller choice must be judged not only by what it can do, but by whether the team can build it, test it, troubleshoot it, and trust it before competition.

Because of this, we kept the EV3 Brick as the main controller. The EV3 was already compatible with our LEGO motors and sensors, so it gave us a stable base for movement, steering, wall following, curve behavior, and parking. Arduino was still useful, but its role was reduced to supporting the HuskyLens vision system instead of controlling the entire robot.

<p align="center">
  <strong>What we learned:</strong><br>
  The best engineering choice is not always the most advanced option. It is the option that can be tested, repaired, and trusted under real constraints.
</p>

## ❀ 2. Code That Became Too Long ────୨ৎ────────୨ৎ────

One of our biggest software failures was assuming that adding more code would automatically make Cheese smarter. Earlier versions of the program reached approximately **1500 lines** because every time the robot failed, we added another condition, exception, or emergency correction.

At first, this seemed like progress. If Cheese crashed near a wall, we added wall protection. If it failed after a curve, we added recovery logic. If a sensor reading was unstable, we added more conditions. However, after many additions, the code became harder to understand and harder to tune.

The real problem was behavior overlap. Several parts of the code could try to control the robot at the same time. For example, PID centering could be trying to correct the robot while curve logic was trying to turn, or wall protection could interrupt another behavior in a way that made the movement unstable. This made it difficult to know which part of the program caused a specific crash.

Our solution was to simplify the program to around **300 lines** and organize it by priorities. Instead of making the robot react to everything at once, the code became clearer about which behavior should control the robot in each situation. This made debugging easier because we could follow the robot’s decision process more clearly.

<p align="center">
  <strong>What we learned:</strong><br>
  More code does not automatically mean better control. For Cheese, shorter code with clearer priorities created more predictable movement than a long code with many overlapping fixes.
</p>

---

## ❀ 3. Infrared Front Sensor in v1 ────୨ৎ────────୨ৎ────

In v1, we used an infrared sensor at the front of the robot because we wanted Cheese to have front detection. The idea was that the robot could use this information to understand what was ahead and react before reaching a problem area.

During testing, the infrared sensor became unreliable because its readings were affected by external light. Shadows, reflections, bright areas, and changes in room lighting could change what the sensor detected. This meant the robot could react to the lighting environment instead of reacting to the actual track.

This was a serious problem because WRO testing conditions are not always identical. A sensor that only works in one lighting setup is risky because the robot may behave differently when the light changes. Since our goal was repeatable performance, we could not depend on a front sensor that changed too much with the environment.

Because of this, we stopped depending on infrared for front detection. This failure helped us understand that sensor selection should not only be based on what a sensor can measure, but also on whether it can measure it consistently under real competition-like conditions.

<p align="center">
  <strong>What we learned:</strong><br>
  A sensor must be tested in the same type of environment where the robot will run. If the readings change too much with lighting, the sensor becomes difficult to trust.
</p>

---

## ❀ 4. Front Ultrasonic Sensor in v2 ────୨ৎ────────୨ৎ────

After the infrared sensor became unreliable, we tried using a front ultrasonic sensor in v2. This seemed like a stronger solution because ultrasonic sensors measure distance directly and are not affected by light in the same way.

However, testing showed that the front ultrasonic sensor created a different problem. It detected too many changes at once, especially near curves and walls. Instead of giving the robot clean and useful information, it sometimes made Cheese react too strongly or at the wrong moment.

This affected the steering. The robot became less precise because the front ultrasonic readings introduced too much noise into the decision process. During curves, the sensor could detect walls or nearby surfaces in a way that made the robot overcorrect, move less smoothly, increase lap time, or crash more often.

This failure helped us realize that a sensor can work correctly and still be wrong for a specific task. The front ultrasonic sensor was not broken, but it was not the best tool for curve behavior. For curves, we needed a cleaner event trigger, not more distance data.

Because of this, we removed the dependency on the front ultrasonic sensor for curve handling. We moved toward a system where side ultrasonic sensors handled wall distance and the color sensor handled curve timing.

<p align="center">
  <strong>What we learned:</strong><br>
  A sensor can be functional but still not useful for a specific job. For Cheese, curve timing needed a clear trigger, while ultrasonic readings were better suited for wall distance.
</p>

## ❀ 5. HuskyLens Recognition Without Extra Lighting ────୨ৎ────────୨ৎ────

The HuskyLens camera was added to support obstacle recognition, but it did not work reliably without extra lighting. The main issue was that the camera did not always see the obstacle colors clearly. During testing, the green pillar sometimes looked almost black, and the red pillar looked brownish or chocolate-colored.

This was a serious failure because the obstacle challenge depends on correct color recognition. If Cheese cannot tell the difference between red and green, it may choose the wrong avoidance direction. In that situation, the robot could have correct movement logic but still fail because the input information from the camera was wrong.

This showed us that camera performance depends heavily on lighting. The HuskyLens was not only limited by its programming or object training; it was also limited by what the camera could physically see. Poor lighting changed the appearance of the objects before the robot even processed them.

After adding lighting support, the obstacle colors appeared closer to their real colors. This made the HuskyLens more reliable because it received a clearer image. The change improved the quality of the sensor input instead of only trying to fix the result through code.

<p align="center">
  <strong>What we learned:</strong><br>
  Vision is not only a software problem. If the camera cannot see the real color of the object, the robot cannot make a reliable decision from that data.
</p>

---

## ❀ 6. Single Light Placement Was Not Enough ────୨ৎ────────୨ৎ────

After adding lighting, we discovered another problem: one light position could not support both the HuskyLens and the color sensor. At first, we thought one lamp would be enough to improve the robot’s sensing conditions. Testing showed that this was too simple.

When the light was placed higher, it helped the HuskyLens because the camera needed better visibility in the front obstacle area. However, that same light did not illuminate the floor enough for the color sensor. The color sensor reads the mat surface very close to the ground, so it needed light in a lower position.

When the light was placed lower, the floor visibility improved, but the HuskyLens did not receive enough support for obstacle recognition. This meant that one light could improve one sensor while leaving the other sensor unstable.

Because of this, we changed to a **dual-light support system**. The upper lamp supports the HuskyLens camera, while the lower lamp supports the color sensor near the floor. This solution added extra wiring and weight, but it solved two different sensing problems with two different lighting positions.

## ❀ 7. Broken Pins and Connection Stress ────୨ৎ────────୨ৎ────

Some Technic pins broke during the build and testing process. This happened either because too much force was used during assembly or because the pins were under pressure in loaded parts of the structure.

At first, broken pins seemed like a small construction issue. However, testing showed that small LEGO parts can become important mechanical failure points. If a pin breaks, loosens, or carries too much stress, the chassis can lose stability. This can affect sensor alignment, steering consistency, and the robot’s ability to repeat the same movement path.

This failure helped us understand that mechanical reliability is not only about motors, wheels, and sensors. The small connection pieces also matter because they hold the full structure together. If the structure changes slightly during testing, the robot may behave differently even with the same code.

Because of this, we started treating pin placement and connection pressure as part of the engineering process. We reinforced critical joints, handled stressed parts more carefully, and paid more attention to which sections carried the most load.

## ❀ 8. Wide Curves and Lack of Fluidity ────୨ৎ────────୨ৎ────

Another failure was that Cheese sometimes took curves too widely. Even when the robot completed the turn, the movement was not always smooth or efficient. A wide curve made the robot travel more distance, lose time, and exit the curve in a less stable position.

This mattered because a curve does not end the moment the robot changes direction. If Cheese exits the curve too close to a wall or at a poor angle, the next straight section becomes harder. The PID and wall protection then have to work harder to recover the robot’s position. This can create more corrections, more oscillation, and more risk of crashing.

The problem was connected to both software and mechanics. The software needed to decide when the curve started, how strongly to steer, and how to recover after the curve. The mechanical steering system also needed to move freely and quickly enough to reach the required angle before the robot reached the wall.

Because of this, curve tuning became a priority. We focused on making curves tighter, improving the transition after the curve, and reducing the amount of correction needed once the robot returned to straight movement.

<p align="center">
  <strong>What we learned:</strong><br>
  Completing a curve is not enough. Cheese must exit the curve in a stable position so the next section of the track starts smoothly.
</p>

---

## ❀ 9. Parking Failure at the First Regional ────୨ৎ────────୨ৎ────

At the first regional, Cheese did not stop where it was supposed to during parking. This affected our final score because the robot lost points for not finishing in the correct area.

This failure showed us that parking cannot be treated as a small detail at the end of the run. The final stop depends on everything that happens before it: lap counting, curve detection, speed control, final alignment, and braking behavior. If any of these parts are inaccurate, the robot may stop too early, too late, or in the wrong position.

Parking became a system-level problem. Cheese needed a reliable way to know when the run was complete, but it also needed to approach the final area with enough control to stop accurately. If the robot reached the final zone too fast, even a correct stop command could be physically too late.

Because of this, parking became its own area for improvement. We began treating the final stop as part of the robot’s scoring performance, not only as the last line of code.

<p align="center">
  <strong>What we learned:</strong><br>
  A successful run is not only about completing laps. Cheese must finish the run with enough control to stop in the correct area.
</p>

## ❀ Final Reflection ────୨ৎ────────୨ৎ────

The most important lesson from these failures is that engineering is not only about choosing components. It is about understanding whether each component, line of code, sensor position, and structural choice works under real testing conditions.

Several systems that seemed useful at first became unreliable on the track. Instead of forcing them to work, we changed the design based on evidence. These failures helped Cheese become more stable, more understandable, and easier to improve.

The final version of Cheese is stronger because it was shaped by problems that we actually observed. The failed Arduino main-control plan helped us protect the project timeline. The long code taught us to simplify logic. The sensor failures taught us to separate sensing tasks. The lighting problems taught us that cameras and color sensors need different support. The mechanical failures taught us that small parts matter. The curve and parking failures taught us that movement quality is just as important as movement completion.

<p align="center">
  <strong>What did not work became the evidence that shaped what finally did.</strong>
</p>
