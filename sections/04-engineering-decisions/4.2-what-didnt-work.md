## ❀ Failure Analysis Method ────୨ৎ────────୨ৎ────

A failed test is useful when it explains what needs to be improved. During the development of Cheese, several ideas looked good in theory but became unreliable once the robot was tested on the track.

We analyzed each failure using the same structure:

<div align="center">

|  Step | Engineering Question     |
| :---: | :----------------------- |
| **1** | What failed?             |
| **2** | Why did it fail?         |
| **3** | What risk did it create? |
| **4** | What did we change?      |
| **5** | What did we learn?       |

</div>

This helped us avoid random changes. Instead of modifying everything at once, we tried to connect every redesign to a specific problem observed during testing.

## ❀ 1. Arduino as the Full Main Controller ────୨ৎ────────୨ৎ────

<p align="center">
  <img src="https://img.shields.io/badge/Failure-Hardware_Integration-red?style=flat-square">
  <img src="https://img.shields.io/badge/Cause-Late_Materials-orange?style=flat-square">
  <img src="https://img.shields.io/badge/Fix-EV3_Main_Control-green?style=flat-square">
</p>

At first, we wanted to use Arduino as the main controller for Cheese. In theory, this would have given us more flexibility with electronics and communication. However, in practice, the materials and integration were not ready early enough to make this reliable for competition.

The problem was not that Arduino was a bad option. The problem was that it became a high-risk option for our timeline. A control system that cannot be fully tested before competition can create more problems than it solves.

<div align="center">

| Option                         | Risk                                                   | Final Decision                                                                   |
| :----------------------------- | :----------------------------------------------------- | :------------------------------------------------------------------------------- |
| **Arduino as full controller** | Required more materials, wiring, and integration time. | Not used as the main controller.                                                 |
| **EV3 as main controller**     | Less flexible than a custom electronics system.        | Chosen because it was stable and already compatible with our motors and sensors. |
| **Arduino Nano as support**    | Still needed careful communication setup.              | Used as a bridge for the HuskyLens vision system.                                |

</div>

**What we learned:** The best engineering choice is not always the most advanced one. It is the one that can be built, tested, repaired, and trusted under real constraints.

## ❀ 2. Code That Became Too Long ────୨ৎ────────୨ৎ────

<p align="center">
  <img src="https://img.shields.io/badge/Failure-Software_Overload-red?style=flat-square">
  <img src="https://img.shields.io/badge/Before-Approx_1500_Lines-orange?style=flat-square">
  <img src="https://img.shields.io/badge/After-Approx_300_Lines-green?style=flat-square">
</p>

One of our biggest software problems was assuming that a longer code would make the robot smarter. Earlier versions reached approximately **1500 lines**, because we kept adding new conditions every time something failed.

Instead of making Cheese more reliable, the extra logic created more behavior conflicts. The robot had too many systems trying to react at the same time, which made it harder to know whether a problem came from PID, curve handling, wall correction, or parking logic.

Our solution was to simplify the code to around **300 lines** and organize it into clearer behavior priorities.

<div align="center">

| Before                             | Problem                         | After                        |
| :--------------------------------- | :------------------------------ | :--------------------------- |
| Approx. **1500 lines**             | Too many overlapping conditions | Approx. **300 lines**        |
| Several behaviors reacting at once | Hard to debug                   | Clearer priority-based logic |
| More code added after each failure | More unpredictable behavior     | More focused tuning          |

</div>

**What we learned:** More code does not automatically mean better control. For Cheese, a shorter and clearer program made the robot easier to understand, tune, and improve.

## ❀ 3. Infrared Front Sensor in v1 ────୨ৎ────────୨ৎ────

<p align="center">
  <img src="https://img.shields.io/badge/v1-Infrared_Front_Sensor-lightgrey?style=flat-square">
  <img src="https://img.shields.io/badge/Failure-Light_Disruption-red?style=flat-square">
</p>

In v1, we used an infrared sensor at the front of the robot. The goal was to give Cheese front detection, but the sensor was affected too much by external light.

This became a problem because the robot does not run in a perfectly controlled lighting environment. Shadows, reflections, bright areas, and external light sources could all change the sensor readings. That made the front detection unreliable.

<div align="center">

| Problem                                     | Risk                                                      |
| :------------------------------------------ | :-------------------------------------------------------- |
| Infrared readings changed with light        | Cheese could react to lighting instead of the real track. |
| Sensor behavior depended on the environment | Less reproducible performance.                            |
| Data was not stable enough                  | Harder to tune the code around it.                        |

</div>

**What we learned:** A sensor must be tested under the same conditions where the robot will actually run. Theoretical usefulness is not enough if the readings are unstable in practice.

## ❀ 4. Front Ultrasonic Sensor in v2 ────୨ৎ────────୨ৎ────

<p align="center">
  <img src="https://img.shields.io/badge/v2-Front_Ultrasonic-blue?style=flat-square">
  <img src="https://img.shields.io/badge/Failure-Steering_Instability-red?style=flat-square">
  <img src="https://img.shields.io/badge/Effect-Crashes_and_Slower_Laps-orange?style=flat-square">
</p>

In v2, we replaced the infrared sensor with a front ultrasonic sensor. This seemed like a stronger option because ultrasonic sensors can measure distance directly. However, during testing, the front ultrasonic sensor detected too many changes at once.

Because of this, the steering became less precise. Instead of following the track smoothly, Cheese reacted too aggressively to incoming readings. The steering sometimes looked uncontrolled, which increased the time between laps and caused the robot to crash more often.

This showed us that the front ultrasonic sensor was not helping the robot during curves. It was adding too much noise into the decision process and making the robot less stable.

<div align="center">

| Cause                                    | Effect on Cheese               | Final Direction                                   |
| :--------------------------------------- | :----------------------------- | :------------------------------------------------ |
| Too many detections at once              | Steering became uncontrolled.  | Remove front ultrasonic dependency.               |
| Readings changed quickly during curves   | The robot became less precise. | Use color sensor for curve timing.                |
| Sensor behavior interfered with steering | More crashes and slower laps.  | Keep ultrasonic sensors focused on wall distance. |

</div>

**What we learned:** A sensor can be functional but still wrong for a specific task. For curves, we needed a cleaner trigger, not more distance information.

## ❀ 5. HuskyLens Recognition Without Extra Lighting ────୨ৎ────────୨ৎ────

<p align="center">
  <img src="https://img.shields.io/badge/System-HuskyLens_Camera-FF69B4?style=flat-square">
  <img src="https://img.shields.io/badge/Failure-Color_Distortion-red?style=flat-square">
  <img src="https://img.shields.io/badge/Fix-Helping_Lamp-green?style=flat-square">
</p>

The HuskyLens camera did not always recognize obstacles reliably without extra lighting. During testing, the obstacle colors appeared too dark or distorted. The green pillar looked almost black, while the red pillar looked brownish or chocolate-colored.

This was a serious issue because the obstacle challenge depends on recognizing red and green pillars correctly. If the camera cannot see the true color of the obstacle, the robot cannot make a safe decision about how to avoid it.

After adding the helping lamp, the camera was able to see the obstacles closer to their real colors. This improved detection and made the vision system more reliable during testing.

<div align="center">

| Without Extra Light                       | With Extra Light                           |
| :---------------------------------------- | :----------------------------------------- |
| Green looked too dark or blackish.        | Green appeared closer to its real color.   |
| Red looked brownish or chocolate-colored. | Red appeared clearer and easier to detect. |
| Obstacle recognition was less reliable.   | Detection became more stable.              |

</div>

**What we learned:** Vision is not only about the camera or code. Lighting conditions directly affect what the robot sees, so lighting became part of the engineering solution.

## ❀ 6. Light Placement Too High ────୨ৎ────────୨ৎ────

<p align="center">
  <img src="https://img.shields.io/badge/Failure-Light_Did_Not_Reach_Floor-red?style=flat-square">
  <img src="https://img.shields.io/badge/Fix-Floor_Level_Lighting-yellow?style=flat-square">
</p>

After adding lighting, we noticed another issue: placing the light high near the camera did not illuminate the floor enough. This mattered because the color sensor reads the track surface, not the same area that the HuskyLens sees.

A light that helps the camera does not automatically help the color sensor. Because of this, we moved additional lighting closer to the ground and near the color sensor. This helped the sensor see the floor more clearly and improved color detection.

<div align="center">

| Sensor           | Original Lighting Problem           | Adjustment                        |
| :--------------- | :---------------------------------- | :-------------------------------- |
| **HuskyLens**    | Needed clearer obstacle visibility. | Helping lamp near camera view.    |
| **Color sensor** | Floor was still not bright enough.  | Light moved closer to the ground. |

</div>

**What we learned:** Each sensor needs lighting based on its target area. Camera lighting and floor-sensor lighting are related, but they are not the same problem.

## ❀ 7. Broken Pins and Connection Stress ────୨ৎ────────୨ৎ────

<p align="center">
  <img src="https://img.shields.io/badge/Failure-Broken_Pins-red?style=flat-square">
  <img src="https://img.shields.io/badge/Cause-Force_and_Pressure-orange?style=flat-square">
  <img src="https://img.shields.io/badge/Fix-Reinforced_Joints-green?style=flat-square">
</p>

Some Technic pins broke during the build and testing process. This happened either because too much force was used while placing them, or because the pins were holding pressure and resistance in parts of the structure.

This failure was important because it showed that small LEGO components can become real mechanical weak points. If a pin is carrying too much stress or is forced into place, the connection may weaken or break. This affects the reliability of the chassis and the repeatability of the robot’s movement.

<div align="center">

| Problem                         | Risk                  | Response                                    |
| :------------------------------ | :-------------------- | :------------------------------------------ |
| Pins broke from excessive force | Weaker joints         | Handle parts more carefully during assembly |
| Pins carried too much pressure  | Structural stress     | Add more support around loaded connections  |
| Connections loosened or failed  | Less reliable chassis | Reinforce critical joints                   |

</div>

**What we learned:** Small connection parts matter. A robot is not only limited by motors and sensors; it is also limited by how much stress its structure can handle.

## ❀ 8. Wide Curves and Lack of Fluidity ────୨ৎ────────୨ৎ────

<p align="center">
  <img src="https://img.shields.io/badge/Failure-Wide_Curves-red?style=flat-square">
  <img src="https://img.shields.io/badge/Effect-Time_Loss-orange?style=flat-square">
  <img src="https://img.shields.io/badge/Fix-Curve_Tuning-blue?style=flat-square">
</p>

Another issue was that Cheese sometimes took curves too widely. Even when the robot completed the turn, the movement was not always fluid. This caused time loss because the robot needed more space and more correction after leaving the curve.

Wide curves also affected consistency. If the robot exited a curve at a poor angle, the next straight section became harder to control. This made the PID and wall protection work harder than necessary.

<div align="center">

| Problem                | Effect                            | Improvement Direction                  |
| :--------------------- | :-------------------------------- | :------------------------------------- |
| Curves were too wide   | More time per lap                 | Tune steering angle and curve behavior |
| Motion lacked fluidity | Less consistent path              | Improve post-curve recovery            |
| Poor exit angle        | More corrections needed afterward | Make curve transitions smoother        |

</div>

**What we learned:** Completing a curve is not enough. The robot also needs to exit the curve in a stable position so the next section can begin smoothly.

## ❀ 9. Parking Failure at the First Regional ────୨ৎ────────୨ৎ────

<p align="center">
  <img src="https://img.shields.io/badge/Failure-Parking_Position-red?style=flat-square">
  <img src="https://img.shields.io/badge/Impact-Points_Lost-orange?style=flat-square">
  <img src="https://img.shields.io/badge/Focus-Final_Stop_Logic-blue?style=flat-square">
</p>

At the first regional, Cheese did not stop where it was supposed to during parking. This affected our final score because the robot lost points for not finishing in the correct area.

This failure showed us that parking could not be treated as a small ending detail. It needed its own logic and testing process. A robot can complete laps successfully, but if it does not stop correctly, the run is still incomplete.

<div align="center">

| What Failed            | Why It Mattered                          | What It Changed                                       |
| :--------------------- | :--------------------------------------- | :---------------------------------------------------- |
| Final parking position | We lost points at the regional.          | Parking became a priority in later logic.             |
| Stop timing/location   | The robot did not finish where expected. | We began testing final stop behavior more carefully.  |
| End-of-run reliability | A good lap was not enough.               | The robot needed a more intentional parking strategy. |

</div>

**What we learned:** The end of the run is part of the engineering challenge. Parking must be designed and tested with the same importance as movement and curve handling.

## ❀ Final Reflection ────୨ৎ────────୨ৎ────

The most important lesson from these failures is that engineering is not only about choosing components. It is about understanding whether each component, line of code, and structural choice works under real testing conditions.

Several systems that seemed useful at first became unreliable on the track. Instead of forcing them to work, we changed the design based on evidence. These failures helped Cheese become more stable, more understandable, and easier to improve.

<p align="center">
  <strong>What did not work became the evidence that shaped what finally did.</strong>
</p>
