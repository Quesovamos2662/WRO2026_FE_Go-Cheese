## ❀ Engineering Philosophy ────୨ৎ────────୨ৎ────

During development, we learned that improving Cheese did not always mean adding more parts or writing more code. Some of our best decisions came from simplifying systems that were creating instability.

Our final design direction became:

<div align="center">

| Main Principle                  | What It Means for Cheese                                                                                   |
| :------------------------------ | :--------------------------------------------------------------------------------------------------------- |
| **Reliability over complexity** | A simpler system that works repeatedly is better than a complex system that is hard to debug.              |
| **Testing over assumptions**    | We changed components only after seeing how they behaved on the real track.                                |
| **Clear subsystem roles**       | Sensors, motors, lights, and code logic each needed a specific purpose.                                    |
| **Rebuildable design**          | Since this is our first WRO Future Engineers season, we needed a robot we could repair and modify quickly. |

</div>

---

## ❀ Decision Overview ────୨ৎ────────୨ৎ────

<div align="center">

| Decision Area            | Original Problem or Idea                                   | Final Decision                                                      | Engineering Benefit                                                            |
| :----------------------- | :--------------------------------------------------------- | :------------------------------------------------------------------ | :----------------------------------------------------------------------------- |
| **Main controller**      | We first wanted to use Arduino as the main controller.     | We kept the EV3 as the main controller.                             | More reliable with our available motors, sensors, materials, and testing time. |
| **Code structure**       | Earlier programs became too long and difficult to control. | We reduced and organized the code into clearer logic.               | Less behavior conflict, easier debugging, and more predictable movement.       |
| **Front sensing system** | Each version needed a different front sensor solution.     | We changed from infrared to ultrasonic to color-based curve timing. | Better matching between sensor type and real track behavior.                   |
| **Camera lighting**      | HuskyLens struggled with inconsistent lighting.            | We added a helping lamp.                                            | More stable obstacle recognition.                                              |
| **Floor lighting**       | Light near the camera did not reach the floor well enough. | We moved extra lighting closer to the ground and color sensor.      | More consistent floor and color detection.                                     |

</div>

## ❀ Decision 1: Keeping EV3 as the Main Controller ────୨ৎ────────୨ৎ────

<p align="center">
  <img src="https://img.shields.io/badge/Decision-EV3_Main_Controller-57C785?style=flat-square">
  <img src="https://img.shields.io/badge/Reason-Material_Limits-FFB347?style=flat-square">
  <img src="https://img.shields.io/badge/Result-More_Reliable_Testing-4A90E2?style=flat-square">
</p>

At first, we considered using Arduino as the main control platform for Cheese. However, because of material limitations and delays in receiving components, we decided to keep the **LEGO EV3 Brick** as the central controller.

This was a practical engineering decision. The EV3 already worked directly with our motors and sensors, which allowed us to focus on testing movement, steering behavior, sensor readings, and software tuning instead of spending too much time solving hardware integration problems.

The Arduino Nano was not removed from the project. Instead, its role changed: it became part of the **vision architecture**, helping connect the HuskyLens camera to the robot while the EV3 remained responsible for navigation and motor control.

<details>
<summary><strong>Why this mattered</strong></summary>

Keeping the EV3 as the main controller reduced risk. Since this is our first season in WRO Future Engineers, we needed a control system that was familiar, stable, and easier to debug under competition pressure. This helped us keep development moving even when some materials arrived late.

</details>

## ❀ Decision 2: Reducing Code Complexity ────୨ৎ────────୨ৎ────

<p align="center">
  <img src="https://img.shields.io/badge/Decision-Code_Simplification-FF69B4?style=flat-square">
  <img src="https://img.shields.io/badge/Problem-Too_Many_Conflicts-red?style=flat-square">
  <img src="https://img.shields.io/badge/Result-Clearer_Logic-blue?style=flat-square">
</p>

One of our biggest software lessons was that a longer code does not automatically make a better robot. Some earlier versions of our program became too extensive because we kept adding logic to fix every new problem. Instead of improving the robot, this sometimes made different behaviors interfere with each other.

We decided to synthesize the code into a clearer structure where each part had a defined job. This made it easier to understand which behavior was controlling the robot at each moment, such as:

<div align="center">

| Behavior            | Purpose                                                                |
| :------------------ | :--------------------------------------------------------------------- |
| **PID centering**   | Keeps Cheese between the walls during normal movement.                 |
| **Wall protection** | Overrides normal movement when the robot gets too close to a wall.     |
| **Curve handling**  | Controls the robot when a curve is detected.                           |
| **Parking logic**   | Helps the robot stop near its starting area after completing the laps. |

</div>

This decision also helped us separate **mechanical problems** from **software problems**. If the robot was unstable, the answer was not always to add more code. Sometimes the better solution was to simplify the program, adjust the sensors, or improve the physical setup.

## ❀ Decision 3: Changing the Front Sensor Strategy Across Versions ────୨ৎ────────୨ৎ────

<p align="center">
  <img src="https://img.shields.io/badge/v1-Infrared_Front_Sensor-lightgrey?style=flat-square">
  <img src="https://img.shields.io/badge/v2-Front_Ultrasonic-blue?style=flat-square">
  <img src="https://img.shields.io/badge/v3-Color_Based_Curve_Timing-yellow?style=flat-square">
</p>

The front sensor system changed across the three versions of Cheese because each version revealed a different weakness during testing.

<div align="center">

| Version | Front Sensor Choice           | What Happened                                                          | What We Learned                                                                            |
| :------ | :---------------------------- | :--------------------------------------------------------------------- | :----------------------------------------------------------------------------------------- |
| **v1**  | Infrared sensor               | Ambient light disrupted the readings too much.                         | The sensor was too sensitive to lighting changes for reliable navigation.                  |
| **v2**  | Front ultrasonic sensor       | It affected curve execution because the readings changed too suddenly. | A front ultrasonic sensor could interfere with curve behavior instead of helping it.       |
| **v3**  | Color sensor for curve timing | Curve events became more directly connected to floor markings.         | The sensor should match the task: wall sensors for walls, color sensor for track markings. |

</div>

In **v1**, the infrared sensor did not work well because external light affected its readings. Since the sensor was too sensitive to lighting conditions, it could not provide stable information for consistent movement.

In **v2**, we changed to a front ultrasonic sensor. This seemed like a stronger option, but it created a new problem: the sensor was too reactive during curve execution. When the robot entered or exited a curve, the ultrasonic readings changed sharply and affected the robot’s behavior.

In **v3**, we moved away from depending on a front ultrasonic sensor for curve behavior. Instead, we relied more on sensors that matched each task: ultrasonic sensors for wall distance and the color sensor for floor-based curve timing.

## ❀ Decision 4: Adding a Helping Lamp for the HuskyLens ────୨ৎ────────୨ৎ────

<p align="center">
  <img src="https://img.shields.io/badge/Decision-External_Lighting-orange?style=flat-square">
  <img src="https://img.shields.io/badge/System-HuskyLens_Camera-FF69B4?style=flat-square">
  <img src="https://img.shields.io/badge/Goal-Safer_Obstacle_Recognition-57C785?style=flat-square">
</p>

We added extra lighting because the HuskyLens camera did not always recognize obstacles reliably under inconsistent lighting conditions. Since the obstacle challenge depends on identifying red and green pillars, poor visibility could cause late or incorrect detections.

The helping lamp was added to make the camera’s view more stable. This decision was important because obstacle recognition is not only a camera problem; it is also a lighting problem. If the image is unclear, the software cannot make a reliable decision from it.

<details>
<summary><strong>Engineering reason</strong></summary>

The HuskyLens needs consistent visual input to recognize objects safely. By improving the lighting environment, we reduced one of the variables that could affect obstacle detection. This makes the camera system more predictable during testing and competition runs.

</details>

## ❀ Decision 5: Moving Light Closer to the Color Sensor ────୨ৎ────────୨ৎ────

<p align="center">
  <img src="https://img.shields.io/badge/Decision-Floor_Level_Lighting-FFD43B?style=flat-square">
  <img src="https://img.shields.io/badge/System-Color_Sensor-4A90E2?style=flat-square">
  <img src="https://img.shields.io/badge/Goal-Consistent_Track_Reading-57C785?style=flat-square">
</p>

At first, placing the helping lamp near the camera seemed like enough. However, we noticed that light placed high on the robot did not illuminate the floor properly. This mattered because the color sensor reads the surface of the track, not the upper field of view.

To solve this, we adapted the lighting placement and added light closer to the ground, near the color sensor. This helped brighten the area where the sensor reads the floor and supported more consistent color detection during movement.

This decision showed us that lighting placement must match the sensor’s job. A light that helps the camera does not automatically help the color sensor. For Cheese, reliable sensing required lighting that supported both obstacle recognition and floor detection.

## ❀ Engineering Impact Summary ────୨ৎ────────୨ৎ────

<div align="center">

| Final Decision                     | Main Impact on Cheese                                          |
| :--------------------------------- | :------------------------------------------------------------- |
| **EV3 as main controller**         | Reduced integration risk and kept motor/sensor control stable. |
| **Shorter, clearer code**          | Made the robot easier to debug, tune, and explain.             |
| **Sensor changes across versions** | Improved the match between sensor behavior and track needs.    |
| **Helping lamp near HuskyLens**    | Improved camera reliability for obstacle detection.            |
| **Lighting near color sensor**     | Improved floor visibility and color detection consistency.     |

</div>

## ❀ Final Reflection ────୨ৎ────────୨ৎ────

The most important pattern in these decisions is that every major change came from testing. We did not keep a component just because it was part of an earlier version, and we did not add complexity just to make the robot look more advanced.

Our final choices focused on making Cheese more reliable, easier to understand, and easier to improve. By keeping the EV3 as the main controller, simplifying the code, changing sensors based on real failures, and improving lighting for both camera and color detection, we created a stronger foundation for the current v3 robot.

<p align="center">
  <strong>Cheese improved because each decision was tested, questioned, and redesigned.</strong>
</p>
