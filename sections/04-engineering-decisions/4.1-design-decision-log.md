## вқҖ Engineering Decision Framework в”Җв”Җв”Җв”ҖаӯЁа§Һв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҖаӯЁа§Һв”Җв”Җв”Җв”Җ

Our design process was built around one main idea: **a reliable system is better than a complicated system that cannot be tested consistently**.

Since this is our first season in the WRO Future Engineers category, we had to make decisions based not only on what was technically possible, but also on what was realistic with our materials, time, experience, and testing conditions.

| Engineering Factor   | How It Affected Cheese                                                                                          |
| :------------------- | :-------------------------------------------------------------------------------------------------------------- |
| **Constraints**      | Limited materials, late component availability, lighting problems, and sensor instability shaped our decisions. |
| **Tradeoffs**        | We often chose reliability and simplicity over more advanced but riskier systems.                               |
| **Iteration**        | v1, v2, and v3 each tested a different sensor and mechanical approach.                                          |
| **Risk mitigation**  | We changed the controller plan, simplified the code, and added lighting to reduce failure points.               |
| **Testing evidence** | Each major change came from something that failed or became unstable during practice.                           |

## вқҖ Decision Overview в”Җв”Җв”Җв”ҖаӯЁа§Һв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҖаӯЁа§Һв”Җв”Җв”Җв”Җ

| Decision Area           | Constraint or Problem                                                                 | Final Decision                                                                                  | Engineering Reason                                                                 |
| :---------------------- | :------------------------------------------------------------------------------------ | :---------------------------------------------------------------------------------------------- | :--------------------------------------------------------------------------------- |
| **Main controller**     | Arduino was considered, but materials arrived late and full integration became risky. | Keep the **EV3 Brick** as the main controller.                                                  | EV3 was already compatible with our motors and sensors, reducing integration risk. |
| **Arduino role**        | Arduino was not ready to control the whole vehicle.                                   | Use Arduino Nano as a **vision bridge** instead.                                                | It could support the HuskyLens without replacing the EV3 control system.           |
| **Code structure**      | Longer code created more behavior conflicts.                                          | Simplify the program into clearer logic layers.                                                 | Easier debugging, tuning, and explanation.                                         |
| **Front sensor system** | Infrared and front ultrasonic sensors caused different reliability issues.            | Use task-specific sensing: side ultrasonic sensors for walls and color sensor for curve timing. | Each sensor is used for the type of data it reads best.                            |
| **Camera visibility**   | HuskyLens recognition changed with lighting conditions.                               | Add a helping lamp.                                                                             | More stable visual input improves obstacle recognition.                            |
| **Floor visibility**    | Light near the camera did not illuminate the floor enough.                            | Add light closer to the ground and color sensor.                                                | Better floor illumination improves color detection.                                |

## вқҖ Decision 1: Keeping EV3 as the Main Controller в”Җв”Җв”Җв”ҖаӯЁа§Һв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҖаӯЁа§Һв”Җв”Җв”Җв”Җ

<p align="center">
  <img src="https://img.shields.io/badge/Constraint-Material_Delay-red?style=flat-square">
  <img src="https://img.shields.io/badge/Decision-EV3_Main_Controller-green?style=flat-square">
  <img src="https://img.shields.io/badge/Tradeoff-Less_Flexibility_More_Reliability-blue?style=flat-square">
</p>

At the beginning, we considered using Arduino as the main control platform for Cheese. This would have given us more flexibility for electronics and communication, but it also depended on having the correct materials, wiring, and integration ready on time.

The constraint was that some materials arrived late, and depending on Arduino as the full control system would have created a major risk close to competition. Because of that, we chose to keep the **LEGO EV3 Brick** as the main controller.

This decision reduced risk because the EV3 already worked directly with our motors and sensors. It allowed us to focus on testing movement, steering, sensor readings, and software behavior instead of spending most of our time solving hardware compatibility issues.

| Option                         | Advantage                               | Risk                                                            |
| :----------------------------- | :-------------------------------------- | :-------------------------------------------------------------- |
| **Arduino as main controller** | More electronics flexibility            | Higher integration risk, more wiring complexity, late materials |
| **EV3 as main controller**     | Direct motor/sensor compatibility       | Less flexible than a custom electronics system                  |
| **Final choice**               | EV3 main control + Arduino support role | Balanced reliability with added vision support                  |

**Final reasoning:** We did not choose EV3 because it was the most advanced option. We chose it because it was the most reliable option for our actual constraints.

## вқҖ Decision 2: Using Arduino Nano as a Vision Bridge в”Җв”Җв”Җв”ҖаӯЁа§Һв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҖаӯЁа§Һв”Җв”Җв”Җв”Җ

<p align="center">
  <img src="https://img.shields.io/badge/System-HuskyLens_+_Arduino_Nano-FF69B4?style=flat-square">
  <img src="https://img.shields.io/badge/Purpose-Vision_Communication-4A90E2?style=flat-square">
</p>

Even though Arduino was not used as the main controller, it still became useful in the robot. We decided to use the **Arduino Nano** as a bridge between the HuskyLens camera and the EV3-based architecture.

This choice allowed us to keep the EV3 focused on navigation and motor control while the Arduino supported the vision system. Instead of forcing one controller to do everything, we separated responsibilities between systems.

| Subsystem            | Controller Role                                |
| :------------------- | :--------------------------------------------- |
| **EV3 Brick**        | Main navigation, motor control, sensor logic   |
| **Arduino Nano**     | Communication bridge for HuskyLens vision data |
| **HuskyLens Camera** | Obstacle recognition support                   |

**Final reasoning:** Arduino was not removed from the project; its role was adjusted to match what it could reliably contribute.

## вқҖ Decision 3: Simplifying the Code в”Җв”Җв”Җв”ҖаӯЁа§Һв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҖаӯЁа§Һв”Җв”Җв”Җв”Җ

<p align="center">
  <img src="https://img.shields.io/badge/Constraint-Code_Complexity-red?style=flat-square">
  <img src="https://img.shields.io/badge/Decision-Simplified_Logic-green?style=flat-square">
  <img src="https://img.shields.io/badge/Result-Easier_Debugging-blue?style=flat-square">
</p>

One of our biggest lessons was that a longer code does not automatically make a better robot. Earlier versions became too extensive because we kept adding more conditions to solve every new problem. Instead of making Cheese smarter, this sometimes created conflicts between behaviors.

For example, one part of the code could try to keep the robot centered while another part was trying to handle a curve. When too many behaviors competed at the same time, the robot became harder to tune and less predictable.

Our decision was to synthesize the code and organize it into clearer control layers.

| Behavior Layer          | Purpose                                                     |
| :---------------------- | :---------------------------------------------------------- |
| **Curve handling**      | Takes priority when the robot is executing a turn.          |
| **Post-curve recovery** | Helps the robot stabilize after leaving a curve.            |
| **Wall protection**     | Prevents the robot from getting too close to the walls.     |
| **PID centering**       | Keeps the robot centered during normal straight movement.   |
| **Parking logic**       | Controls the final stop after completing the required laps. |

**Final reasoning:** We chose clearer logic over longer logic. This made the robot easier to debug, explain, and improve.

## вқҖ Decision 4: Changing the Front Sensor Strategy Across Versions в”Җв”Җв”Җв”ҖаӯЁа§Һв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҖаӯЁа§Һв”Җв”Җв”Җв”Җ

<p align="center">
  <img src="https://img.shields.io/badge/v1-Infrared_Front_Sensor-lightgrey?style=flat-square">
  <img src="https://img.shields.io/badge/v2-Front_Ultrasonic-blue?style=flat-square">
  <img src="https://img.shields.io/badge/v3-Color_Based_Curve_Timing-yellow?style=flat-square">
</p>

The front sensor strategy changed across the three versions of Cheese because each version revealed a different limitation.

| Version | Front Sensor Choice           | Problem Found                                                      | Resulting Decision                                     |
| :------ | :---------------------------- | :----------------------------------------------------------------- | :----------------------------------------------------- |
| **v1**  | Infrared sensor               | Readings were disrupted by external light.                         | Stop depending on infrared for front detection.        |
| **v2**  | Front ultrasonic sensor       | It reacted too sharply during curves and affected curve execution. | Remove front ultrasonic dependency for curve behavior. |
| **v3**  | Color sensor for curve timing | Better matched to floor markings.                                  | Use color detection for curve events.                  |

In **v1**, the infrared sensor was too sensitive to ambient light. Since the competition environment can include shadows, reflections, and inconsistent lighting, the infrared readings were not stable enough.

In **v2**, we replaced it with a front ultrasonic sensor. This solved one problem but created another: during curves, the robotвҖҷs angle changed quickly, which made the front ultrasonic readings unstable. The sensor was too reactive and could interfere with curve execution.

In **v3**, we changed the logic so the robot uses sensors according to their strongest purpose. Side ultrasonic sensors are used for wall distance, while the color sensor is used for floor-based curve timing.

**Final reasoning:** We did not choose sensors based only on what they could measure. We chose them based on whether their readings stayed useful during the robotвҖҷs real movement.

## вқҖ Decision 5: Adding a Helping Lamp for the HuskyLens в”Җв”Җв”Җв”ҖаӯЁа§Һв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҖаӯЁа§Һв”Җв”Җв”Җв”Җ

<p align="center">
  <img src="https://img.shields.io/badge/Constraint-Unstable_Lighting-orange?style=flat-square">
  <img src="https://img.shields.io/badge/Decision-Helping_Lamp-green?style=flat-square">
  <img src="https://img.shields.io/badge/Goal-Safer_Recognition-blue?style=flat-square">
</p>

The HuskyLens camera needs a clear and consistent view to recognize obstacles. During testing, we noticed that lighting conditions affected how safely the camera could detect objects. Since the obstacle challenge depends on recognizing pillars correctly, inconsistent vision became a risk.

We added a helping lamp to improve the cameraвҖҷs input. This decision was not just about making the robot brighter; it was about reducing one of the variables that could cause vision failure.

| Problem               | Risk                                            | Mitigation                                       |
| :-------------------- | :---------------------------------------------- | :----------------------------------------------- |
| Inconsistent lighting | Camera may detect obstacles late or incorrectly | Add a helping lamp near the vision system        |
| Shadows or dark areas | Object recognition becomes less stable          | Improve visibility in the cameraвҖҷs field of view |

**Final reasoning:** Vision reliability depends on lighting. The helping lamp was added to make obstacle recognition safer and more repeatable.

## вқҖ Decision 6: Moving Light Closer to the Color Sensor в”Җв”Җв”Җв”ҖаӯЁа§Һв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҖаӯЁа§Һв”Җв”Җв”Җв”Җ

<p align="center">
  <img src="https://img.shields.io/badge/Constraint-Floor_Not_Bright_Enough-red?style=flat-square">
  <img src="https://img.shields.io/badge/Decision-Floor_Level_Lighting-yellow?style=flat-square">
  <img src="https://img.shields.io/badge/System-Color_Sensor-blue?style=flat-square">
</p>

At first, placing light near the HuskyLens seemed like enough. However, we realized that light placed higher on the robot did not illuminate the floor properly. This mattered because the color sensor reads the track surface, not the upper view seen by the camera.

Because of that, we adapted the lighting position and placed additional light closer to the ground and near the color sensor. This improved the area where the sensor reads floor colors and helped make curve detection more consistent.

| Sensor               | What It Needs           | Lighting Solution          |
| :------------------- | :---------------------- | :------------------------- |
| **HuskyLens Camera** | Clear view of obstacles | Light near the camera area |
| **Color Sensor**     | Clear view of the floor | Light closer to the ground |

**Final reasoning:** A lighting solution must match the sensor it supports. One lamp position cannot always solve both camera recognition and floor detection.

## вқҖ Final Engineering Impact в”Җв”Җв”Җв”ҖаӯЁа§Һв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҖаӯЁа§Һв”Җв”Җв”Җв”Җ

| Final Decision                     | Constraint Addressed                        | Impact on Cheese                      |
| :--------------------------------- | :------------------------------------------ | :------------------------------------ |
| **EV3 as main controller**         | Material delays and integration risk        | More stable development path          |
| **Arduino as vision bridge**       | Need for HuskyLens communication            | Supports vision without replacing EV3 |
| **Simplified code**                | Behavior conflicts and debugging difficulty | More predictable control logic        |
| **Sensor changes across versions** | Unstable front sensing                      | Better match between sensor and task  |
| **Helping lamp**                   | Camera recognition affected by lighting     | Safer obstacle visibility             |
| **Floor-level lighting**           | Color sensor needed better illumination     | More consistent floor detection       |

---

## вқҖ Final Reflection в”Җв”Җв”Җв”ҖаӯЁа§Һв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҖаӯЁа§Һв”Җв”Җв”Җв”Җ

The strongest design decisions in Cheese came from testing, not guessing. We did not keep a system just because it was part of an earlier version, and we did not add complexity only to make the robot seem more advanced.

Our final choices focused on making Cheese more reliable, easier to debug, and easier to explain. By keeping the EV3 as the main controller, simplifying the code, changing sensors based on real failures, and improving lighting for both the camera and color sensor, we created a stronger foundation for the v3 robot.
