## ❀ Engineering Decision Framework ────୨ৎ────────୨ৎ────

Our design process was guided by one main principle: **a reliable robot is better than a complicated robot that cannot be tested consistently**. Since this was our first season in the WRO Future Engineers category, we had to make decisions that matched our actual experience, materials, testing time, and competition pressure.

Many design ideas looked useful at first, but testing showed that not every idea was realistic for Cheese. Some options added too much wiring, some made the code harder to debug, and others created new mechanical or sensing problems. Because of this, we learned to judge each decision by asking whether it made the robot more stable, easier to repair, easier to tune, or easier to explain.

For each major decision, we considered four questions: **What problem were we trying to solve? What did testing show? What tradeoff did the decision create? How did the decision improve Cheese?** This helped us avoid adding features only because they looked advanced. Instead, we focused on changes that improved the robot’s real performance on the mat.

The front section of Cheese became the most important area for engineering decisions because it contains the systems that affect direction, sensing, and reaction time. The steering mechanism determines how quickly the robot can change direction, the color sensor helps detect curve events, the HuskyLens supports obstacle recognition, and the lighting system improves the reliability of both camera and floor detection.

Because all of these systems are close to each other, a change in one area often affected another. For example, changing the sensor position affected detection timing, adding lights affected weight and wiring, and increasing speed affected whether the steering system had enough time to react. This is why our final v3 layout was not only about placing components where they fit physically; it was about placing them where they could work together reliably.

## ❀ Decision 1: Keeping EV3 as the Main Controller ────୨ৎ────────୨ৎ────

At the beginning of the project, we considered using Arduino as the main controller for Cheese. This idea seemed useful because Arduino gives more flexibility for external modules, custom wiring, and electronics expansion. It could have made the robot feel more advanced, especially for camera communication and non-LEGO components.

However, using Arduino as the main controller also created a major risk. Some materials arrived late, and a full Arduino-based robot would need extra time for wiring, motor control, sensor communication, debugging, and safety testing. If the main control system failed close to competition, the entire robot would become harder to fix because both the movement system and the sensing system would depend on a setup we had not fully validated.

Because of this, we decided to keep the **LEGO EV3 Brick** as the main controller. This was not the most flexible option, but it was the safest option for our timeline. The EV3 already worked directly with our LEGO motors and sensors, which allowed us to focus on movement logic, wall following, curve handling, and parking instead of rebuilding the entire control architecture.

The tradeoff was that EV3 limited some electronic flexibility. However, the benefit was greater because it gave us a stable base for testing. When Cheese failed, we could focus on the robot’s behavior instead of constantly questioning whether the controller, wiring, or communication system was also causing the problem.

<p align="center">
  <strong>Final reasoning:</strong><br>
  We kept the EV3 as the main controller because it reduced integration risk and gave Cheese a stable control foundation during a season with limited time and changing materials.
</p>

---

## ❀ Decision 2: Using Arduino Nano as a Vision Bridge ────୨ৎ────────୨ৎ────

Even though we did not use Arduino as the main controller, we did not remove it from the project completely. Instead, we changed its role. The Arduino Nano became a support component for the vision system, working as a bridge between the HuskyLens camera and the EV3-based robot architecture.

This decision helped us separate responsibilities. The EV3 remained responsible for the main driving behavior: motor control, steering, ultrasonic readings, color sensor readings, PID correction, curve logic, and final stopping. The Arduino Nano supported the part that EV3 could not handle as easily: communication with the HuskyLens camera.

This separation made the robot easier to understand. Instead of forcing one system to do everything, each component had a clearer job. The EV3 controlled movement, while the Arduino Nano helped bring vision data into the robot’s decision-making process. This made the HuskyLens more useful without making the whole robot depend on Arduino.

The tradeoff was that adding Arduino Nano increased wiring complexity. However, this was still less risky than making Arduino control the entire robot. By giving Arduino a smaller and more specific role, we could benefit from it without letting it become a major failure point.

<p align="center">
  <strong>Final reasoning:</strong><br>
  Arduino was not rejected; its role was adjusted. It became useful as a vision bridge because that task matched what it could support reliably within our final system.
</p>

## ❀ Decision 3: Simplifying the Code ────୨ৎ────────୨ৎ────

One of the biggest lessons from Cheese was that a longer code does not automatically create a better robot. Earlier versions of the program reached approximately **1500 lines** because we kept adding new conditions, special cases, and emergency fixes every time the robot crashed or behaved incorrectly.

At first, adding more code seemed like the right solution. If the robot had a problem near a wall, we added a wall condition. If it failed after a curve, we added a recovery condition. If it missed a color, we added another detection rule. However, after many changes, the code became harder to control because several behaviors could try to act at the same time.

For example, the robot could be trying to center itself using PID while another part of the code was trying to handle a curve or protect the robot from a wall. When behaviors competed with each other, Cheese became less predictable. The issue was not that each piece of code was wrong by itself; the issue was that the full system became too difficult to tune.

Our final decision was to reduce the code to a clearer structure of approximately **300 lines**. Instead of using many disconnected fixes, we organized the robot around behavior priorities. Curve handling, wall protection, PID centering, post-curve recovery, and parking logic each had a clearer role. This made testing easier because we could understand which behavior was controlling the robot at each moment.

<p align="center">
  <strong>Final reasoning:</strong><br>
  We simplified the code because predictable logic is more valuable than long logic. A shorter program with clear priorities made Cheese easier to debug, tune, and explain.
</p>

## ❀ Decision 4: Changing the Front Sensor Strategy Across Versions ────୨ৎ────────୨ৎ────

The front sensor strategy changed across v1, v2, and v3 because each version revealed a different limitation. This was not a random redesign. Each change came from a specific testing problem that affected the robot’s ability to move consistently.

In **v1**, we used an infrared sensor in the front. The problem was that infrared readings were too affected by external lighting. Shadows, reflections, and changes in room light could make the readings unstable. Since WRO testing conditions are not always perfectly controlled, this made the sensor risky for a robot that needed repeatable decisions.

In **v2**, we replaced the infrared sensor with a front ultrasonic sensor. This solved the light sensitivity problem, but it created a different issue. The front ultrasonic sensor detected many changes at once, especially near curves and walls. Instead of giving clean information, it sometimes made Cheese react too strongly or at the wrong moment. This made steering less precise, increased lap time, and caused more crashes.

In **v3**, we changed the strategy completely. Instead of expecting one front sensor to solve every navigation problem, we gave each sensor a more specific job. The side ultrasonic sensors were used for wall distance and centering, while the color sensor was used for curve timing based on floor colors. This made the sensing system more organized because each sensor was used for the type of information it could read best.

</p>

<p align="center">

This decision improved reliability because it reduced confusing sensor behavior. The robot no longer depended on a front ultrasonic sensor to interpret complex curve situations. Instead, curve events were linked to color detection, while wall correction stayed connected to the ultrasonic sensors.

<p align="center">
  <strong>Final reasoning:</strong><br>
  We changed the front sensing strategy because each sensor needed a specific purpose. Cheese became more reliable when wall distance, curve timing, and obstacle recognition were separated into different sensing tasks.
</p>

## ❀ Decision 5: Adding a Dual-Light Support System ────୨ৎ────────୨ৎ────

Lighting became a major design decision because both the HuskyLens camera and the color sensor were affected by visibility. However, they did not need the same type of light. The HuskyLens needed clearer frontal lighting to recognize obstacle pillars, while the color sensor needed light closer to the floor to detect track colors consistently.

At first, we thought one lamp could support the whole sensing system. Testing showed that this was not enough. When the lamp was placed higher, the HuskyLens view improved, but the floor near the color sensor was still not bright enough. When the lamp was placed lower, the color sensor improved, but the camera did not receive enough light to detect obstacles reliably.

Because of this, we added a **dual-light support system**. The upper lamp supports the HuskyLens by improving obstacle visibility, while the lower lamp supports the color sensor by illuminating the floor. Both lights are powered by a separate 9V battery, which prevents the EV3 rechargeable battery from carrying the extra lighting load.

<p align="center">
  <img src="../../v-photos/v3/dual_light_system_v3.jpeg" alt="Cheese v3 dual-light support system" width="75%">
</p>

<p align="center">
  <em>Cheese v3 dual-light support system. The upper lamp helps the HuskyLens camera see the obstacle area more clearly, while the lower lamp improves floor illumination near the color sensor.</em>
</p>

The tradeoff was that the robot gained extra wiring, extra weight, and another battery to manage. However, this added complexity was justified because unstable sensing caused bigger problems than the lights themselves. If the robot cannot read the environment correctly, even a good program can make the wrong decision.

<p align="center">
  <strong>Final reasoning:</strong><br>
  We used two lights because one light position could not support both camera recognition and floor color detection. The dual-light system gave each sensor the type of illumination it needed.
</p>

## ❀ Decision 6: Improving Obstacle Color Recognition ────୨ৎ────────୨ৎ────

Obstacle recognition is important because the robot must react differently depending on whether it sees a red or green pillar. If the HuskyLens misreads the obstacle color, Cheese can choose the wrong path and lose control during the obstacle challenge.

During testing, we noticed that the obstacle colors did not always appear correctly through the camera. The green obstacle sometimes looked almost black, while the red obstacle looked brownish or chocolate-colored. This was not only a camera problem; it was a lighting problem. The camera could only classify what it could see, and poor lighting changed how the colors appeared.

The decision to improve lighting was therefore connected directly to obstacle strategy. We were not adding lights only to make the robot look better. We added them because the vision system needed a more stable visual input. After lighting support was added, the obstacles appeared closer to their real colors, which made detection more reliable.

This decision also showed us that software accuracy depends on sensor input quality. A program can be correctly written, but if the camera receives poor visual information, the robot will still make poor decisions. Improving the physical sensing conditions made the software more useful.

<p align="center">
  <strong>Final reasoning:</strong><br>
  We improved obstacle color recognition by improving the camera’s environment. Better lighting reduced color distortion and gave the HuskyLens a more reliable image to process.
</p>

## ❀ Decision 7: Treating Mechanical Stress as a Design Constraint ────୨ৎ────────୨ৎ────

Mechanical reliability became an important part of the design process because Cheese is not only a software project. The robot must physically survive repeated testing, starts, corrections, impacts, and handling. During building and testing, some Technic pins broke because of excess force or because they were under too much pressure in loaded parts of the structure.

This failure showed us that small LEGO components can become important mechanical failure points. A broken pin can weaken the chassis, change alignment, or create looseness in parts that need to stay stable. If the structure shifts during movement, the sensors and steering system can also become less consistent.

Because of this, we started treating pin placement and connection pressure as part of the engineering process. We paid more attention to which areas carried load, which connections were under stress, and which parts needed reinforcement. This helped us understand that a strong robot is not only made from strong motors, but also from stable connections.

<p align="center">

The tradeoff was that reinforcing the structure sometimes added weight or made the design less compact. However, stability was more important than having the lightest possible chassis. A slightly heavier but stronger robot is more useful than a lighter robot that loses alignment during testing.

<p align="center">
  <strong>Final reasoning:</strong><br>
  We treated mechanical stress as a design constraint because even small structural failures can affect steering, sensing, and overall reliability.
</p>

## ❀ Decision 8: Prioritizing Tighter Curves and Smoother Movement ────୨ৎ────────୨ৎ────

Curve handling became one of the most important movement decisions because the robot does not only need to turn; it needs to turn in a way that supports the next section of the run. During early testing, some curves were too wide or not fluid enough. Even when Cheese completed the turn, it sometimes exited at a poor angle and needed extra correction afterward.

This caused two problems. First, wide curves increased lap time because the robot traveled a less efficient path. Second, unstable curve exits made the next straight section harder because the robot started it too close to a wall or at the wrong angle. This meant that a curve failure could continue affecting the robot even after the turn was finished.

Because of this, we treated curve behavior as both a software and mechanical decision. The software needed to control when the curve started, how strongly the robot steered, and how it recovered afterward. At the same time, the mechanical steering system needed to move freely enough to reach the required angle before the robot reached the wall.

The tradeoff was between speed and control. A faster robot could finish laps sooner, but it also gave the steering system less time to reach the correct angle. A slower robot was easier to control, but it could lose time. Our goal was not simply to make Cheese faster; it was to make the robot’s path smoother and more repeatable.

<p align="center">
  <strong>Final reasoning:</strong><br>
  We prioritized tighter and smoother curves because a successful turn is not only about changing direction. It is about exiting the curve in a stable position that supports the next movement.
</p>

---

## ❀ Decision 9: Improving Parking Reliability ────୨ৎ────────୨ৎ────

Parking became a separate design concern because completing the laps is not enough if the robot cannot stop in the correct area. During testing, we learned that the final stop depends on the entire run, not only on the last command. If curve counting, timing, speed, or final alignment is slightly off, the robot can stop too early, too late, or outside the intended zone.

This made parking a system-level problem. The robot needed reliable curve or lap information before it could know when to begin the final stopping process. It also needed controlled speed so the stop could happen accurately. If Cheese reached the final area too fast, even a correct stop command could fail because the robot would continue moving during the braking moment.

Because of this, we connected parking reliability to the same ideas used in the rest of the robot: sensor reliability, speed management, and clear behavior priority. Parking logic needed to activate only after the correct run condition was reached, and it needed to control the robot in a way that improved final accuracy instead of depending on luck.

The tradeoff was that parking logic could not be too complicated. If it depended on too many fragile conditions, it could fail the same way earlier long code versions failed. The final goal was to make parking specific enough to be accurate, but simple enough to test repeatedly.

<p align="center">
  <strong>Final reasoning:</strong><br>
  We improved parking reliability because the final stop is part of the robot’s scoring performance. Cheese must not only complete the laps, but finish them with enough control to stop in the correct area.
</p>

## ❀ Final Engineering Impact ────୨ৎ────────୨ৎ────

The final design of Cheese was shaped by decisions that connected mechanical structure, software behavior, sensing, and testing. Keeping EV3 as the main controller gave the robot a stable control base. Using Arduino Nano as a bridge allowed us to include HuskyLens vision without making the entire robot depend on Arduino. Simplifying the code reduced behavior conflicts and made debugging more realistic.

The sensor changes improved navigation because each sensor was assigned a clearer purpose. Side ultrasonic sensors support wall distance, the color sensor supports curve timing, and the HuskyLens supports obstacle recognition. The dual-light system improved the conditions for both the camera and the color sensor, making the robot less dependent on external lighting.

Mechanical changes also affected performance. Broken pins and stressed connections showed us that structure matters as much as motors and sensors. A loose or weak chassis can affect sensor alignment, steering consistency, and repeated testing reliability. Curve tuning and parking refinement then connected the mechanical and software sides of the robot, because both depended on controlled speed and predictable steering response.

---

## ❀ Final Reflection ────୨ৎ────────୨ৎ────

The strongest design decisions in Cheese came from testing, not guessing. We did not keep a system only because it was part of an earlier version, and we did not add complexity only to make the robot look more advanced. Each major change came from a problem we observed during building, testing, or competition preparation.

The main lesson was that every subsystem affects another subsystem. The controller affects how easily the code can be tested. The code affects how predictable the movement becomes. The sensors affect when the robot reacts. The lighting affects whether the sensors read correctly. The structure affects whether the sensors and steering stay aligned. The speed affects whether the steering has enough time to respond.

By keeping the EV3 as the main controller, simplifying the code, changing sensors based on real failures, improving lighting for both the camera and color sensor, reinforcing mechanical weak points, improving curve behavior, and refining parking logic, we created a stronger foundation for Cheese v3.

<p align="center">
  <strong>Cheese improved because every major decision was connected to a real constraint, tradeoff, or test result.</strong>
</p>
