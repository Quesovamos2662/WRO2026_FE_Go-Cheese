## ❀ Engineering Decision Framework ────୨ৎ────────୨ৎ────

Our design process was built around one main idea: **a reliable system is better than a complicated system that cannot be tested consistently**.

Since this is our first season in the WRO Future Engineers category, we had to make decisions based on more than just what looked advanced. We also had to consider what was realistic with our materials, time, programming experience, lighting conditions, testing results, and competition schedule.

For this reason, every major decision followed the same engineering pattern:

<div align="center">

| Step | Question We Asked | Why It Mattered |
| :--- | :--- | :--- |
| **1. Problem** | What was failing or becoming unreliable? | We needed to identify the real cause instead of guessing. |
| **2. Evidence** | What did testing show? | Decisions had to come from observed robot behavior. |
| **3. Tradeoff** | What did we gain and what did we lose? | Every change improved one area but usually affected another. |
| **4. Final Decision** | What option gave the safest result? | We prioritized reliability under competition conditions. |
| **5. Impact** | How did the robot improve after the change? | Each decision needed to make Cheese easier to control, debug, or reproduce. |

</div>

This framework helped us avoid adding random features. Instead, we focused on decisions that made Cheese more stable, more understandable, and more realistic to improve before competition.

## ❀ Decision 1: Keeping EV3 as the Main Controller ────୨ৎ────────୨ৎ────

<p align="center">
  <img src="https://img.shields.io/badge/Constraint-Material_Delay-red?style=flat-square">
  <img src="https://img.shields.io/badge/Decision-EV3_Main_Controller-green?style=flat-square">
  <img src="https://img.shields.io/badge/Tradeoff-Less_Flexibility_More_Reliability-blue?style=flat-square">
</p>

At the beginning of the project, we considered using Arduino as the main controller for Cheese. This idea was attractive because Arduino gives more flexibility for electronics, custom wiring, and external modules. However, making Arduino the main control system would also require more integration work.

The main problem was not that Arduino was a bad option. The problem was that using Arduino as the main controller became too risky for our timeline. Some materials arrived late, and a full Arduino-based control system would need extra wiring, motor control setup, sensor communication, testing, and troubleshooting before becoming competition-ready.

Because of this, we chose to keep the **LEGO EV3 Brick** as the main controller. The EV3 was already compatible with our motors and LEGO sensors, which allowed us to focus on navigation, steering, curve behavior, and parking instead of rebuilding the entire control architecture.

<div align="center">

| Option | Advantage | Risk |
| :--- | :--- | :--- |
| **Arduino as main controller** | More electronics flexibility and customization. | Higher integration risk, more wiring complexity, and less testing time. |
| **EV3 as main controller** | Direct compatibility with LEGO motors and sensors. | Less flexible than a custom electronics system. |
| **Final choice** | EV3 as the main control system. | Best balance between reliability and available time. |

</div>

### Engineering Impact

Keeping the EV3 as the main controller made Cheese easier to test and debug. When the robot failed, we could focus on the behavior itself instead of questioning whether the hardware communication was also failing. This decision reduced uncertainty during testing.

<p align="center">
  <strong>Final reasoning:</strong><br>
  We did not choose EV3 because it was the most advanced option. We chose it because it was the most reliable option for our actual constraints.
</p>

## ❀ Decision 2: Using Arduino Nano as a Vision Bridge ────୨ৎ────────୨ৎ────

<p align="center">
  <img src="https://img.shields.io/badge/System-HuskyLens_+_Arduino_Nano-FF69B4?style=flat-square">
  <img src="https://img.shields.io/badge/Purpose-Vision_Communication-4A90E2?style=flat-square">
  <img src="https://img.shields.io/badge/Architecture-Separated_Roles-green?style=flat-square">
</p>

Even though Arduino was not used as the main controller, it still became useful in the final robot. Instead of removing Arduino completely, we changed its role. The Arduino Nano became a communication bridge between the **HuskyLens camera** and the EV3-based control system.

This decision helped us separate responsibilities. The EV3 focuses on movement, motor control, ultrasonic readings, color sensor readings, and main decision logic. The Arduino Nano supports the vision system by helping manage the HuskyLens communication.

<div align="center">

| Component | Role in the Final Architecture |
| :--- | :--- |
| **EV3 Brick** | Main navigation, motor control, sensor logic, and driving behavior. |
| **Arduino Nano** | Communication bridge for HuskyLens vision data. |
| **HuskyLens Camera** | Obstacle recognition support for red and green pillars. |

</div>

### Tradeoff

Using Arduino Nano as a bridge added another electronic component to the robot, which increased wiring complexity slightly. However, it avoided the much larger risk of making Arduino responsible for the entire robot. This made the system more realistic to test and maintain.

### Engineering Impact

This decision kept the project flexible without making it unstable. We could still use the HuskyLens vision system, but the EV3 remained responsible for the most important driving decisions.

<p align="center">
  <strong>Final reasoning:</strong><br>
  Arduino was not removed from the project; its role was adjusted to match what it could reliably contribute.
</p>

## ❀ Decision 3: Simplifying the Code ────୨ৎ────────୨ৎ────

<p align="center">
  <img src="https://img.shields.io/badge/Constraint-Code_Complexity-red?style=flat-square">
  <img src="https://img.shields.io/badge/Before-Approx_1500_Lines-orange?style=flat-square">
  <img src="https://img.shields.io/badge/After-Approx_300_Lines-green?style=flat-square">
</p>

One of our biggest software lessons was that a longer code does not automatically make a better robot. Earlier versions of Cheese’s program reached approximately **1500 lines** because we kept adding new conditions, exceptions, and fixes every time the robot failed.

At first, this seemed useful because each new condition was meant to solve a specific problem. However, over time, the code became harder to understand and harder to tune. Some behaviors started interfering with each other. For example, the robot could be trying to center itself with PID while another part of the code was trying to react to a curve or correct away from a wall.

This created behavior conflicts. The robot was not only reacting to the environment; it was also reacting to competing instructions inside the code.

<div align="center">

| Problem in Earlier Code | Effect on Cheese |
| :--- | :--- |
| Too many conditions | Harder to know which behavior controlled the robot. |
| Multiple behaviors active at once | Robot movement became less predictable. |
| Long debugging process | Small changes could affect unrelated parts of the program. |
| Repeated fixes added quickly | Some fixes solved one problem but created another. |

</div>

Our decision was to synthesize the code into a clearer structure of approximately **300 lines**. Instead of having many disconnected fixes, the simplified code uses clearer behavior layers with priorities.

<div align="center">

| Behavior Layer | Purpose | Priority Reason |
| :--- | :--- | :--- |
| **Curve handling** | Controls the robot during a confirmed curve. | Curves require specific steering and override normal movement. |
| **Post-curve recovery** | Stabilizes the robot after leaving a curve. | Prevents unstable exits from causing crashes. |
| **Wall protection** | Reacts when the robot gets too close to a wall. | Prevents immediate collisions. |
| **PID centering** | Keeps the robot centered during normal straight movement. | Used when no higher-priority behavior is active. |
| **Parking logic** | Controls the final stop after the required laps. | Finishing position matters for scoring. |

</div>

### Engineering Impact

The simplified code made Cheese easier to debug because each behavior had a clearer purpose. It also made the robot easier to explain in documentation, since the logic could be described as a layered system instead of a long list of exceptions.

<p align="center">
  <strong>Final reasoning:</strong><br>
  We chose clearer logic over longer logic because predictable behavior is more valuable than code that tries to solve every problem at once.
</p>

## ❀ Decision 4: Changing the Front Sensor Strategy Across Versions ────୨ৎ────────୨ৎ────

<p align="center">
  <img src="https://img.shields.io/badge/v1-Infrared_Front_Sensor-lightgrey?style=flat-square">
  <img src="https://img.shields.io/badge/v2-Front_Ultrasonic-blue?style=flat-square">
  <img src="https://img.shields.io/badge/v3-Color_Based_Curve_Timing-yellow?style=flat-square">
</p>

The front sensor strategy changed across the three versions of Cheese because each version revealed a different limitation. This was not a random redesign. It was a response to how each sensor behaved under real testing conditions.

In **v1**, we used an infrared sensor. The problem was that infrared readings were affected by external light. Since the WRO environment can include shadows, reflections, and inconsistent lighting, the sensor was not reliable enough for a robot that needed repeatable behavior.

In **v2**, we moved to a front ultrasonic sensor. This solved part of the light problem, but created a new issue. The front ultrasonic sensor detected too much at once, especially near curves and walls. Instead of making the robot more precise, it sometimes made the steering unstable because the robot reacted to distance changes that were not useful for the current movement.

In **v3**, we changed the strategy. Instead of expecting one front sensor to solve every navigation problem, we separated the sensing tasks:

<div align="center">

| Sensor | Final Purpose |
| :--- | :--- |
| **Side ultrasonic sensors** | Measure wall distance and support centering. |
| **Color sensor** | Detect floor colors used for curve timing. |
| **HuskyLens camera** | Support obstacle recognition. |

</div>

<p align="center">
  <img src="../../v-photos/v3/front_v3.jpg" alt="Cheese v3 front sensor placement" width="75%">
</p>

<p align="center">
  <em>Cheese v3 front view showing the final physical layout after changing the front sensing strategy.</em>
</p>

<div align="center">

| Version | Sensor Strategy | Problem Found | What Changed |
| :--- | :--- | :--- | :--- |
| **v1** | Infrared front sensing | Affected by lighting conditions. | Stopped depending on infrared for front detection. |
| **v2** | Front ultrasonic sensing | Detected too much at once and made steering unstable. | Removed front ultrasonic dependency for curve behavior. |
| **v3** | Task-specific sensors | Needed clearer separation of sensor roles. | Used side ultrasonic sensors for walls and color sensor for curves. |

</div>

### Engineering Impact

This decision improved reliability because each sensor was used for the type of information it reads best. The side ultrasonic sensors are better for wall distance, while the color sensor is better for detecting floor-based curve events. This reduced confusion in the robot’s logic.

<p align="center">
  <strong>Final reasoning:</strong><br>
  We did not choose sensors only because they could measure something. We chose them based on whether their readings stayed useful during real robot movement.
</p>

## ❀ Decision 5: Adding a Dual-Light Support System ────୨ৎ────────୨ৎ────

<p align="center">
  <img src="https://img.shields.io/badge/Constraint-Unstable_Lighting-orange?style=flat-square">
  <img src="https://img.shields.io/badge/Decision-Dual_Light_System-green?style=flat-square">
  <img src="https://img.shields.io/badge/Goal-Safer_Sensing-blue?style=flat-square">
</p>

Lighting became one of the most important sensing problems on Cheese. During testing, we noticed that the HuskyLens camera and the color sensor were both affected by lighting, but they needed light in different places.

The HuskyLens needed stronger visibility in the front area so it could recognize the obstacle pillars. The color sensor needed light close to the floor so it could detect the track colors more consistently. At first, we thought one lamp could solve both problems. Testing showed that this was not true.

When the light was placed higher, the camera view improved, but the floor near the color sensor was still not illuminated enough. When the light was placed lower, the color sensor improved, but the camera did not receive enough support for obstacle visibility.

Because of this, Cheese uses a **dual-light support system** powered by a separate 9V battery.

<p align="center">
  <img src="../../v-photos/v3/dual_light_system_v3.jpeg" alt="Cheese v3 dual-light support system" width="75%">
</p>

<p align="center">
  <em>Cheese v3 dual-light support system. The upper lamp helps the HuskyLens camera see the obstacle area more clearly, while the lower lamp improves floor illumination near the color sensor.</em>
</p>

<div align="center">

| Lighting Element | Position | Sensor Supported | Purpose |
| :--- | :--- | :--- | :--- |
| **Upper helping lamp** | Upper front area | HuskyLens camera | Improves obstacle visibility and helps the camera see pillar colors more accurately. |
| **Lower helping lamp** | Lower front area, near the floor | Color sensor | Improves floor illumination for more consistent track color detection. |
| **External 9V battery** | Separate lighting circuit | Both lamps | Powers the lights without draining the EV3 rechargeable battery. |

</div>

### Tradeoff

The dual-light system adds extra weight, wiring, and battery management. However, the benefit was greater than the cost because unstable sensing caused bigger problems than the added complexity.

### Engineering Impact

The dual-light system made sensing more consistent. It helped the HuskyLens see obstacles more clearly and helped the color sensor read floor colors with less dependence on ambient light.

<p align="center">
  <strong>Final reasoning:</strong><br>
  A lighting solution must match the sensor it supports. One lamp position could not solve both camera recognition and floor detection, so we separated the lighting into two support points.
</p>

## ❀ Decision 6: Improving Obstacle Color Recognition ────୨ৎ────────୨ৎ────

<p align="center">
  <img src="https://img.shields.io/badge/System-HuskyLens_Camera-FF69B4?style=flat-square">
  <img src="https://img.shields.io/badge/Problem-Color_Distortion-red?style=flat-square">
  <img src="https://img.shields.io/badge/Result-Real_Color_View-green?style=flat-square">
</p>

Obstacle recognition is critical because the robot must react differently depending on the obstacle color. If the HuskyLens misreads red or green, the robot can make the wrong movement decision.

During testing, the obstacle colors did not always appear correctly through the camera. The green obstacle sometimes looked almost black, while the red obstacle appeared brownish or chocolate-colored. This made detection less reliable because the camera was not seeing the colors close enough to their real appearance.

The lighting system improved this problem by making the colors appear clearer and more consistent.

<div align="center">

| Before Lighting Support | Risk | After Dual-Light Support |
| :--- | :--- | :--- |
| Green looked blackish. | Camera could fail to recognize the green obstacle correctly. | Green appeared closer to its real color. |
| Red looked brownish or chocolate-colored. | Camera could confuse or ignore the red obstacle. | Red appeared clearer and easier to detect. |
| Recognition changed with lighting. | Robot behavior became less predictable. | Detection became more stable during testing. |

</div>

### Engineering Impact

This decision showed that camera performance depends on the environment, not only on the code. Even if the program is correct, poor lighting can make the sensor data incorrect. By improving the camera’s visual input, the robot had a better chance of making the correct obstacle decision.

<p align="center">
  <strong>Final reasoning:</strong><br>
  Vision reliability depends on lighting quality. The dual-light system reduced color distortion and helped the robot make safer obstacle decisions.
</p>

## ❀ Decision 7: Treating Mechanical Stress as a Design Constraint ────୨ৎ────────୨ৎ────

<p align="center">
  <img src="https://img.shields.io/badge/Constraint-Mechanical_Stress-red?style=flat-square">
  <img src="https://img.shields.io/badge/Problem-Broken_Pins-orange?style=flat-square">
  <img src="https://img.shields.io/badge/Decision-Reinforced_Connections-green?style=flat-square">
</p>

Mechanical reliability is not only about motors and sensors. During building and testing, some Technic pins broke because of excess force during assembly or because they were under too much pressure in loaded parts of the structure.

This showed us that small LEGO components can become important failure points. A broken pin can weaken a joint, change alignment, or make the chassis less stable. If the structure becomes loose, the sensors and steering system can also lose consistency.

<p align="center">
  <img src="../../img/broken_pin.jpeg" alt="Broken Technic pin from Cheese testing" width="55%">
</p>

<p align="center">
  <em>Broken Technic pin caused by excess force and structural pressure during testing. This helped us treat connection stress as a real mechanical design constraint.</em>
</p>

<div align="center">

| Mechanical Issue | Risk | Design Response |
| :--- | :--- | :--- |
| Pins broke from excessive force | Weaker joints and less stable structure. | Handle parts more carefully during assembly. |
| Pins carried too much pressure | Higher chance of structural failure. | Reinforce loaded connections. |
| Connections could loosen during testing | Sensor alignment and steering consistency could change. | Improve joint support in critical areas. |

</div>

### Engineering Impact

This decision helped us treat structural strength as part of robot performance. If the chassis is not stable, even good sensors and code can become unreliable because the robot’s physical alignment changes during movement.

<p align="center">
  <strong>Final reasoning:</strong><br>
  A robot is only as reliable as its weakest connection. Reinforcing small structural points helped make Cheese more durable during repeated testing.
</p>

## ❀ Decision 8: Prioritizing Tighter Curves and Smoother Movement ────୨ৎ────────୨ৎ────

<p align="center">
  <img src="https://img.shields.io/badge/Constraint-Time_Loss-orange?style=flat-square">
  <img src="https://img.shields.io/badge/Problem-Wide_Curves-red?style=flat-square">
  <img src="https://img.shields.io/badge/Decision-Curve_Tuning-blue?style=flat-square">
</p>

Another important decision was improving how Cheese handled curves. Earlier testing showed that wide curves caused time loss and made the robot less fluid between sections of the track.

A curve was not considered successful only because the robot turned. It also needed to exit the curve in a stable position. If Cheese exited too wide or at a poor angle, it needed extra correction afterward. This made the next straight section more unstable and increased the risk of hitting a wall.

<div align="center">

| Curve Problem | Effect on Run | Engineering Response |
| :--- | :--- | :--- |
| Curves were too wide | More distance traveled and more time lost. | Tune curve steering behavior. |
| Movement lacked fluidity | Robot needed more corrections after turns. | Improve transition after curves. |
| Poor exit angle | Higher risk of wall contact after the curve. | Add or tune post-curve recovery. |
| High approach speed | Steering had less time to reach the needed angle. | Use speed control during correction-heavy moments. |

</div>

### Engineering Impact

This decision connected mechanical movement and software tuning. The robot needed enough steering angle to complete the curve, but also enough speed control to avoid entering the curve too aggressively. A faster robot is not useful if it exits curves in a position that causes crashes or extra corrections.

<p align="center">
  <strong>Final reasoning:</strong><br>
  Completing a curve is not enough. The robot must complete the curve in a way that supports the next movement section.
</p>

## ❀ Decision 9: Improving Parking Reliability ────୨ৎ────────୨ৎ────

<p align="center">
  <img src="https://img.shields.io/badge/Challenge-Final_Stop-orange?style=flat-square">
  <img src="https://img.shields.io/badge/Problem-Parking_Accuracy-red?style=flat-square">
  <img src="https://img.shields.io/badge/Goal-Return_To_Start-green?style=flat-square">
</p>

Parking became an important design problem because the robot does not only need to complete laps. It also needs to stop in the correct area after finishing the run. During testing and competition experience, we observed that finishing the movement was not enough if the robot stopped too early, too late, or in the wrong row.

This problem was connected to both sensing and timing. The robot needed a reliable way to know when the required laps were completed, but it also needed enough control to slow down and stop accurately.

<div align="center">

| Parking Issue | Why It Happened | Design Direction |
| :--- | :--- | :--- |
| Robot stopped too early | Final trigger happened before reaching the intended area. | Improve final stop conditions. |
| Robot stopped too late | Timing or position reference was not accurate enough. | Add more reliable parking logic. |
| Robot reached the final zone too fast | Braking distance became harder to control. | Slow down before final stop. |
| Curve counting affected parking | Wrong lap or curve count changed the final trigger. | Improve curve counting and final logic connection. |

</div>

### Engineering Impact

This decision helped us understand that parking is not a separate task at the end of the run. It depends on the quality of the entire run: curve counting, speed control, sensor reliability, and final braking. A small error earlier in the run can affect where the robot stops at the end.

<p align="center">
  <strong>Final reasoning:</strong><br>
  Parking accuracy depends on both navigation and final speed control. Cheese must not only complete the laps, but also finish them in a way that allows a controlled stop.
</p>

## ❀ Final Reflection ────୨ৎ────────୨ৎ────

The strongest design decisions in Cheese came from testing, not guessing. We did not keep a system just because it was part of an earlier version, and we did not add complexity only to make the robot seem more advanced.

The main lesson was that every subsystem affects another subsystem. The controller affects how easily the code can be tested. The code affects how predictable the movement becomes. The sensors affect when the robot reacts. The lighting affects whether the sensors read correctly. The structure affects whether the sensors and steering stay aligned. The speed affects whether the steering has enough time to respond.

Our final choices focused on making Cheese more reliable, easier to debug, and easier to explain. By keeping the EV3 as the main controller, simplifying the code, changing sensors based on real failures, improving lighting for both the camera and color sensor, reinforcing mechanical weak points, and improving movement behavior, we created a stronger foundation for the v3 robot.

<p align="center">
  <strong>Cheese improved because every major decision was connected to a real constraint, tradeoff, or test result.</strong>
</p>
