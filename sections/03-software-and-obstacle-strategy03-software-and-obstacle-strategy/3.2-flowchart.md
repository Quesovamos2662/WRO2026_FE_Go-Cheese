## ❀ Flowchart Purpose ────୨ৎ────────୨ৎ────

The purpose of this flowchart is to show how Cheese makes decisions during a run. Instead of representing the program as only lines of code, the diagram separates the robot’s behavior into clear logic blocks. This makes it easier to understand which actions happen first, which behaviors have priority, and how the robot changes from normal driving to curve handling or parking.

The flowchart is important because Cheese does not follow only one simple movement rule. The robot must constantly decide between several possible behaviors: staying centered between the walls, protecting itself from collisions, confirming floor colors, entering or exiting curves, recovering after a curve, counting laps, and preparing for the final stop. If these behaviors are not organized by priority, they can interfere with each other and make the robot unpredictable.

This diagram shows that Cheese’s software is structured as a **priority-based control system**. The robot first checks the most important conditions, such as valid sensor readings and color events, before continuing with normal PID driving. This helps prevent the robot from treating every situation the same way.

---

## ❀ 1. Initialization Logic ────୨ৎ────────୨ৎ────

The flowchart begins with the initialization stage. In this part, Cheese prepares all the systems needed before movement starts. The motors, ultrasonic sensors, color sensor, display, steering motor, constants, counters, and state variables are initialized.

This step is necessary because the robot needs a known starting state before it can make reliable decisions. If old values remain from a previous run, the robot could count curves incorrectly, start with the wrong steering position, or activate a behavior that should still be inactive.

The initialization also resets the main run-tracking variables. For example, the completed curve counter and completed lap counter begin at zero. The lap position references are also cleared, so the robot does not use old parking data from a previous test.

```text
completed_curves = 0
completed_laps = 0
lap1_position = None
lap2_position = None
```

The robot also determines the **starting row** by checking whether it is closer to the left or right wall. This matters because the final parking logic later depends on understanding the robot’s original row reference.

---

## ❀ 2. Row Alignment Before Starting ────୨ৎ────────୨ৎ────

Before beginning the run, Cheese performs a row alignment routine. The robot reads both ultrasonic sensors several times and determines which wall is closer. This gives the robot a basic understanding of whether it is starting closer to the left row or the right row.

If the robot is not centered correctly for that starting row, it slowly adjusts by turning toward the farther wall. This helps Cheese begin the run from a more stable position instead of starting at a poor angle.

This step is important because a bad starting alignment can affect the whole run. If Cheese begins too close to one wall, the PID controller may need to correct strongly immediately. That can create early oscillation, unstable steering, or a bad approach into the first curve.

The row alignment also resets the PID values after positioning. This prevents old correction values from affecting the first movement.

---

## ❀ 3. Start Movement ────୨ৎ────────୨ৎ────

After the robot finishes alignment, it begins movement. At this point, Cheese records the start time, sets the drive speed to the normal straight speed, and centers the steering at `0°`.

This starting block creates the first active movement state of the robot. The start time is useful because later logic can compare the robot’s current time or position to previous references. The straight speed gives Cheese enough movement to begin the run, while the centered steering prevents the robot from entering the track with an unnecessary turn.

```text
record start_time
set drive speed = STRAIGHT_SPEED
set steering = 0°
```

This stage connects setup to active navigation. After this point, Cheese enters the main loop, where sensor readings and decision logic repeat continuously.

---

## ❀ 4. Main Loop Structure ────୨ৎ────────୨ৎ────

The main loop is the core of the robot’s behavior. It repeats continuously until the parking logic becomes active or the robot needs to stop. Every loop cycle follows the same general structure:

```text
read sensors
check if readings are valid
check color detection
choose behavior
update movement
repeat
```

This structure is important because Cheese needs to react constantly. The robot cannot decide its full path only once at the start. Wall distances, color readings, steering state, and curve state can change every moment, so the program must keep checking and updating the robot’s behavior.

The loop also includes a small delay. This delay prevents the program from reading sensors too aggressively while still allowing the robot to react quickly enough during movement.

---

## ❀ 5. Sensor Reading and Validity Check ────୨ৎ────────୨ৎ────

Inside the main loop, Cheese reads the left ultrasonic sensor, right ultrasonic sensor, and color sensor. These three inputs decide most of the robot’s movement.

The ultrasonic sensors provide distance information from the walls. This is used for PID centering and wall protection. The color sensor detects floor colors that indicate curve events.

Before using the ultrasonic values, the flowchart checks whether the wall readings are valid. This is necessary because sensors can sometimes return incorrect, missing, or unstable values. If the robot receives invalid wall readings, it cannot safely calculate its position between the walls.

When the readings are invalid, Cheese stops to avoid continuing with unreliable information. This is a safety decision. It is better for the robot to stop than to drive blindly with false sensor data.

---

## ❀ 6. Normal Driving with PID and Wall Protection ────୨ৎ────────୨ৎ────

If no color is detected, Cheese continues with normal driving. This is where the PID centering system and protected wall logic work together.

The PID controller compares the left and right ultrasonic distances. If the robot is centered, the difference between the two readings should be small. If one side is closer than the other, the error becomes larger, and the robot steers to correct its position.

```text
error = left_distance - right_distance
derivative = error - last_error
output = (KP × error) + (KI × integral) + (KD × derivative)
```

The steering output is then limited so the robot does not turn too aggressively. This prevents extreme steering values from making Cheese unstable.

However, PID is not the only protection system. The flowchart also includes wall protection. If either wall is too close, Cheese reduces speed and steers away from the danger side. There are two levels of wall protection:

```text
gentle wall correction
severe wall correction
```

Gentle correction is used when the robot is getting close to a wall but is not in immediate danger. Severe correction is used when the robot is much closer and needs a stronger reaction. This layered protection helps Cheese stay between the walls while avoiding sudden crashes.

---

## ❀ 7. Color Confirmation Logic ────୨ৎ────────୨ৎ────

When the color sensor detects a possible color, Cheese does not immediately accept it. Instead, the flowchart sends the reading through a color confirmation process.

This is important because the color sensor can sometimes detect noise, shadows, reflections, or brief incorrect readings. If the robot accepted every single color reading instantly, it could start or finish curves at the wrong time.

The confirmation logic uses a candidate color and a count. If the same color appears repeatedly, the count increases. If a different color appears, the candidate is reset.

```text
candidate_color = detected color
candidate_count += 1
```

Only when the candidate count reaches the required number of confirmation readings does the robot accept the color. This makes color detection more stable because the robot reacts only to confirmed colors, not random flashes.

---

## ❀ 8. Accepted Color Handling ────୨ৎ────────୨ৎ────

After a color is accepted, Cheese sends it to the accepted color handler. This block decides whether the color means the start of a new curve or the exit of an active curve.

The logic compares the accepted color with the last accepted color. If the accepted color is the first color of a new curve, the robot starts curve mode. If the accepted color is the opposite color while already in a curve, the robot exits the curve.

This makes the curve logic depend on color sequence instead of only distance. The first accepted color begins the curve, and the second different color confirms that the curve has ended.

---

## ❀ 9. Curve Entry Logic ────୨ৎ────────୨ৎ────

When Cheese detects the first color of a new curve, it activates curve mode.

```text
curve_active = True
curve_direction = based on color
drive speed = CURVE_SPEED
steering = CURVE_ANGLE
```

During curve mode, the robot’s normal PID behavior is not the main controller. This is important because PID centering can fight against curve steering. If the robot tried to center itself while also trying to curve, the steering commands could conflict.

The flowchart shows that curve handling has its own steering angle and speed. This gives Cheese a stronger and more intentional curve movement instead of relying on normal wall-following behavior.

The curve direction depends on which color was detected first. This allows the robot to decide whether the curve should be handled toward one side or the other.

---

## ❀ 10. Curve Exit and Curve Counting ────୨ৎ────────୨ৎ────

When Cheese detects the opposite color, the flowchart treats this as the curve exit. At this moment, curve mode is turned off, the completed curve counter increases, and the PID system is reset.

```text
curve_active = False
completed_curves += 1
reset_pid()
```

Resetting PID after a curve is important because the robot’s position and angle have changed. If old PID values continued after the curve, the robot could overcorrect when returning to normal driving.

After the curve exit, Cheese also activates post-curve recovery. This recovery phase helps the robot stabilize before returning fully to normal PID and wall protection.

---

## ❀ 11. Lap Counting Logic ────୨ৎ────────୨ৎ────

The flowchart counts completed curves and uses that value to determine when a lap is complete. When the number of completed curves reaches the required number of curves per lap, the lap counter increases.

```text
completed_laps += 1
completed_curves = 0
```

This is important because the robot does not simply run for a fixed time. It tracks its progress through curve events. After each completed lap, Cheese can record position references. These references are later used by the final stop logic.

The lap counter also prevents the robot from starting parking too early. Parking should only begin after the robot has completed the required number of laps.

---

## ❀ 12. Final Stop Initiation ────୨ৎ────────୨ৎ────

After Cheese reaches the required number of laps, the flowchart initiates the final stop logic. This block calculates parking-related references such as first color delay, first color distance, lap length, stop deadline, and stop position target.

The purpose of this block is to prepare the robot for parking instead of stopping randomly. Cheese needs to estimate where it should stop based on references recorded earlier in the run.

```text
parking_active = True
```

Once parking becomes active, the robot leaves the normal navigation structure and begins checking final stop conditions.

---

## ❀ 13. Parking and Final Stop Logic ────୨ৎ────────୨ৎ────

When parking logic is active, Cheese checks whether it has reached the stop deadline or the stop position target. If either condition is reached, the robot stops.

After stopping, the flowchart shows that Cheese turns toward the starting row and drives forward slowly until it becomes aligned with that row. Then it stops again and ends the program.

This final parking logic is important because finishing the laps is not enough. The robot must also stop in a controlled position. If Cheese reaches the final zone too fast, stops too late, or uses the wrong reference, it can lose parking accuracy.

The flowchart treats parking as its own behavior instead of a small command at the end. This makes the final stop more intentional and easier to test.

---

## ❀ 14. Post-Curve Recovery Logic ────୨ৎ────────୨ৎ────

Post-curve recovery activates after Cheese exits a curve. The purpose of this logic is to stabilize the robot before it returns to normal PID driving.

During a curve, the robot may exit at an angle that is not perfectly centered. If PID immediately takes control with full correction, the robot may overreact. Post-curve recovery helps smooth this transition by using a controlled steering direction, recovery speed, and recovery time.

```text
post_curve_active = True
post_curve_end_time = now + POST_CURVE_TIME
```

When the recovery time ends, post-curve mode turns off, PID is reset, and the robot resumes normal wall protection and centering. This is important because it prevents the robot from carrying unstable curve behavior into the next straight section.

---

## ❀ Behavior Priority Explanation ────୨ৎ────────୨ৎ────

The flowchart shows that Cheese does not treat every behavior equally. Some behaviors must override others because they are more urgent or more specific.

| Priority | Behavior | Reason |
| :---: | :--- | :--- |
| **1** | Invalid sensor stop | The robot should not continue if important readings are unreliable. |
| **2** | Parking / final stop | After the required laps, finishing correctly becomes the main goal. |
| **3** | Curve handling | Curves require specific steering that overrides normal PID. |
| **4** | Post-curve recovery | The robot needs to stabilize after a turn before returning to normal driving. |
| **5** | Wall protection | Prevents crashes when the robot gets too close to a wall. |
| **6** | PID centering | Handles normal straight movement when no higher-priority behavior is active. |

This priority system helps prevent behavior conflicts. For example, curve handling should not compete with PID centering, and parking should not be interrupted by normal driving logic once the final stop has started.

## ❀ Engineering Importance of the Flowchart ────୨ৎ────────୨ৎ────

This flowchart is important because it shows the connection between software logic and robot behavior. Cheese’s movement depends on sensors, timing, counters, steering angles, and speed changes. If these systems are not organized, the robot can behave unpredictably.

The diagram also shows that the robot’s logic was designed through layers. Normal PID driving handles the basic movement, wall protection handles danger situations, color confirmation prevents false curve triggers, curve logic controls turns, post-curve recovery stabilizes exits, and parking logic controls the final stop.

This makes the program easier to debug. If Cheese fails during a straight section, we can focus on PID and ultrasonic readings. If it fails during a curve, we can check color confirmation and curve handling. If it fails after a curve, we can check post-curve recovery. If it fails at the end, we can test parking logic separately.

## ❀ Flowchart Summary ────୨ৎ────────୨ৎ────

The flowchart represents Cheese’s full navigation cycle. The robot begins by initializing its motors, sensors, constants, counters, and state variables. Before starting movement, it checks its row alignment using the ultrasonic sensors so it can begin from a more stable position.

Once the run begins, Cheese enters a continuous main loop. In each cycle, it reads the ultrasonic sensors and color sensor, checks whether the readings are valid, and chooses the correct behavior. During normal movement, the robot uses PID centering and protected wall logic to stay between the walls. When a valid color is confirmed, the robot enters or exits curve mode depending on whether the color begins a new curve or completes the current one.

After each completed curve, Cheese updates the curve counter and uses that information to count laps. Once the required number of laps is completed, the robot starts the final stop logic. At that point, parking becomes the main behavior, and Cheese uses recorded timing and position references to stop and align with its starting row.
