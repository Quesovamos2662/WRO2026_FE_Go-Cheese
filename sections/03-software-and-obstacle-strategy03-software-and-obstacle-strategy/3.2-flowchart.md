## ❀ Flowchart Purpose ────୨ৎ────────୨ৎ────

The purpose of this flowchart is to show how Cheese makes decisions during a full run. Instead of explaining the robot only through code, the flowchart separates the program into clear behavior blocks. This makes it easier to understand what happens before the run starts, what happens inside the main loop, and how the robot changes from normal navigation to curve handling and final parking.

Cheese cannot use one single behavior for the whole track. During the run, the robot must stay between two walls, correct itself when it gets too close to one side, detect floor colors, confirm whether a color reading is real, count curves, track laps, recover after turns, and stop after completing the required number of laps.

The flowchart shows that Cheese uses a **priority-based control structure**. This means that the robot does not treat every behavior equally. Some behaviors, such as curve handling or final parking, must override normal PID driving because they require more specific control.

---

## ❀ 1. Initialization Logic ────୨ৎ────────୨ৎ────

The flowchart begins with the initialization stage. In this part, Cheese prepares all systems before movement starts. The motors, ultrasonic sensors, color sensor, display, steering motor, constants, counters, and state variables are initialized.

The robot also clears all previous run information so the new attempt starts from a clean state. This is important because old values from a previous test could make the robot count curves incorrectly, classify the lap wrong, or activate parking too early.

```text
completed_curves = 0
completed_laps = 0
lap1_position = None
lap2_position = None
```

The main hardware used in this flowchart is:

| Component | Purpose |
| :--- | :--- |
| **EV3 Large Motor** | Provides rear-wheel propulsion. |
| **EV3 Medium Motor** | Controls the Ackermann steering system. |
| **Left ultrasonic sensor** | Measures distance from the left wall. |
| **Right ultrasonic sensor** | Measures distance from the right wall. |
| **Color sensor** | Detects floor colors used for curve logic and lap counting. |
| **EV3 display and buttons** | Show startup messages and allow run control. |

This stage is necessary because Cheese needs a known starting state before the main loop begins. If steering, counters, or sensor modes are not reset correctly, the robot may behave unpredictably even if the rest of the code is correct.

---

## ❀ 2. Row Alignment Before Starting ────୨ৎ────────୨ৎ────

Before starting normal movement, Cheese performs a row alignment routine. The robot reads both wall distances using the left and right ultrasonic sensors. These readings help determine whether Cheese is starting closer to the left or right wall.

If the robot is not aligned with the intended row, it slowly turns toward the farther wall until the distances become more balanced or until the alignment timeout is reached. This step helps the robot begin the run from a cleaner and more stable position.

This matters because the first seconds of the run affect the rest of the lap. If Cheese begins too close to a wall or at a poor angle, the PID controller may need to make strong corrections immediately. That can cause early oscillation, unstable steering, or a bad entry into the first curve.

After row alignment, the PID values are reset. This prevents the robot from carrying old correction values into the actual run.

---

## ❀ 3. Start Movement ────୨ৎ────────୨ৎ────

After initialization and alignment, Cheese begins movement. The robot records the start time, sets the drive speed to the normal straight speed, and centers the steering at `0°`.

```text
record start_time
set drive speed = STRAIGHT_SPEED
set steering = 0°
```

This block connects setup to active navigation. The start time becomes useful later because the robot can compare run timing and movement references when preparing for the final stop. Centering the steering at the start also helps prevent Cheese from entering the first straight section with an unwanted turn.

Once this stage is complete, Cheese enters the main loop.

---

## ❀ 4. Main Loop Structure ────୨ৎ────────୨ৎ────

The main loop is the central part of Cheese’s software. It repeats continuously until the robot reaches the final parking condition or the program is stopped manually.

Each cycle follows the same general order:

```text
read sensors
validate wall readings
check color detection
confirm color if needed
choose movement behavior
update counters and states
repeat
```

This structure allows Cheese to react during the run instead of depending only on fixed timing. The robot keeps checking its wall distance, floor color readings, curve state, lap progress, and parking condition. This makes the system more adaptable because the robot can respond to what is happening on the track.

---

## ❀ 5. Sensor Reading and Validity Check ────୨ৎ────────୨ৎ────

Inside the main loop, Cheese reads the left ultrasonic sensor, right ultrasonic sensor, and color sensor.

The ultrasonic sensors provide wall distance information. These readings are used for PID centering and wall protection. The color sensor detects floor colors that are used for curve handling and lap counting.

Before using wall readings, the robot checks whether they are valid. This is important because ultrasonic sensors can sometimes return missing or incorrect values. If the robot used a false reading, it could steer in the wrong direction and crash.

If the wall readings are invalid or unsafe to use, the flowchart includes a stop condition. This is a safety behavior because it is better for Cheese to stop than to continue driving with unreliable wall data.

---

## ❀ 6. Normal Driving with PID and Wall Protection ────୨ৎ────────୨ৎ────

If no confirmed color event is active, Cheese continues with normal driving. In this mode, the robot uses ultrasonic-based PID centering to stay between the walls.

The PID controller compares the left and right wall distances. If both readings are balanced, the robot is close to the center. If one side is closer, the error increases and Cheese turns to correct its position.

```text
error = left_distance - right_distance
derivative = error - last_error
output = (KP × error) + (KI × integral) + (KD × derivative)
```

The output is limited to a maximum steering angle so the robot does not overcorrect. This helps reduce sudden movements and makes straight driving smoother.

Wall protection works as a safety layer above normal PID. If Cheese gets too close to a wall, the robot reduces speed and steers away from the danger side. There are two correction levels:

| Protection Level | Purpose |
| :--- | :--- |
| **Gentle wall correction** | Used when Cheese is getting close to a wall but is not in immediate danger. |
| **Severe wall correction** | Used when Cheese is too close and needs a stronger correction. |

This layered design is important because PID is good for normal centering, but wall protection is better for urgent situations. If Cheese is close to a wall, avoiding the crash becomes more important than staying perfectly centered.

---

## ❀ 7. Color Confirmation Logic ────୨ৎ────────୨ৎ────

When the color sensor detects a possible floor color, Cheese does not immediately accept it. Instead, the reading passes through a confirmation process.

This is important because a single color reading can be affected by shadows, reflections, sensor height, or small lighting changes. If the robot accepted every color instantly, it could count false curves or change behavior at the wrong moment.

The confirmation logic uses a candidate color and a counter. If the same color appears repeatedly, the counter increases. If a different color appears, the candidate resets.

```text
candidate_color = detected_color
candidate_count += 1
```

Only after the same color is detected enough times does Cheese accept it as a real event. This makes the color logic more reliable because the robot reacts to stable color readings instead of random flashes.

---

## ❀ 8. Accepted Color Handling ────୨ৎ────────୨ৎ────

After a color is accepted, Cheese sends it to the accepted color handler. This part of the logic decides whether the color represents the start of a new curve or the end of the current curve.

The curve system is based on a blue/orange pairing idea. One stable color starts the curve sequence, and the opposite stable color completes it. This prevents the robot from counting one color twice as a full curve.

```text
first color detected → start curve sequence
opposite color detected → complete curve
```

This logic is important because the Open Challenge requires the robot to complete multiple curves and laps. By using confirmed color pairs, Cheese can track progress through the course without depending only on time.

---

## ❀ 9. Curve Entry Logic ────୨ৎ────────୨ৎ────

When Cheese accepts the first color of a new curve, the robot marks that a curve sequence has started.

```text
curve_active = True
required_exit_color = opposite color
```

The robot stores the curve start reference, such as current time or current motor position. This helps the program know when the curve sequence began and prevents false duplicate events from being counted immediately.

In the flowchart, curve entry is shown as a special behavior because curve sections require different logic from normal straight movement. Normal PID tries to keep the robot centered between walls, while curve logic focuses on detecting and completing a turn event.

Depending on the exact code version, the accepted color may directly trigger a curve steering behavior or may only track the curve-counting state while PID and wall protection continue controlling movement. In both cases, the important idea is the same: the first confirmed color starts the curve sequence and prepares the robot to wait for the opposite color.

---

## ❀ 10. Curve Exit and Curve Counting ────୨ৎ────────୨ৎ────

When Cheese detects the opposite confirmed color, the robot treats this as the end of the curve sequence. At that moment, the curve is counted as completed.

```text
completed_curves += 1
curve_active = False
reset_pid()
```

The PID reset is important because the robot’s position and angle may have changed during the curve. If old PID values continue after a curve, Cheese may overcorrect when returning to normal driving.

The curve counter is one of the most important progress-tracking variables in the flowchart. Instead of relying only on elapsed time, Cheese uses curve completion to estimate where it is in the run.

To avoid false curve counts, the logic can reject events that happen too close together or too quickly. This prevents sensor flicker or duplicate readings from being counted as separate curves.

---

## ❀ 11. Lap Counting Logic ────୨ৎ────────୨ৎ────

The flowchart groups curves into laps. Since each lap contains a fixed number of curves, Cheese can update the lap counter after the required number of curves has been completed.

```text
if completed_curves >= CURVES_PER_LAP:
    completed_laps += 1
    completed_curves = 0
```

Lap counting is important because parking should not activate before the robot has completed the required number of laps. If Cheese starts the final stop too early, it may stop in the wrong place. If it starts too late, it may continue into an extra lap.

The flowchart also shows that the robot can record movement references after completed laps. These references can help estimate lap length and support the final parking logic.

---

## ❀ 12. Final Stop Initiation ────୨ৎ────────୨ৎ────

After Cheese completes the required number of laps, the robot initiates the final stop logic. At this point, the robot’s goal changes. It no longer focuses on completing another lap; it focuses on finishing the run in a controlled position.

The flowchart uses recorded run references to estimate when the robot should stop. These references can include:

| Reference | Purpose |
| :--- | :--- |
| **Start time** | Helps compare current run timing with the beginning of the run. |
| **First color delay** | Estimates the time from start to the first curve marker. |
| **Lap position references** | Help estimate how far the robot travels per lap. |
| **Lap length estimate** | Supports prediction of the final stopping position. |
| **Stop deadline or stop target** | Defines when the robot should stop or begin parking behavior. |

This is a software-based parking strategy. It does not depend on a rear ultrasonic sensor. Instead, it uses the robot’s internal progress information, such as lap completion, motor position, and timing references.

```text
parking_active = True
```

Once parking becomes active, final stopping becomes the priority behavior.

---

## ❀ 13. Parking and Final Stop Logic ────୨ৎ────────୨ৎ────

When parking logic is active, Cheese checks whether it has reached the stop deadline or stop target. If the condition is reached, the robot stops.

The flowchart also includes a controlled alignment step after the initial stop. This represents the idea that the robot should not simply stop anywhere after three laps. It should finish as close as possible to the intended starting row or final area.

This part of the logic is important because the final stop is part of the competition performance. A robot can complete the laps correctly and still lose points if it finishes in the wrong position. For this reason, parking is treated as its own behavior rather than only a final `stop()` command.

The parking logic should be tested separately from the rest of the run. If the robot completes laps but misses the final stop, the issue may be in the stop reference, lap count, final approach speed, or recorded timing.

---

## ❀ 14. Post-Curve Recovery Logic ────୨ৎ────────୨ৎ────

Post-curve recovery activates after Cheese exits a curve. The purpose is to help the robot stabilize before returning fully to normal PID and wall protection.

During a curve, the robot may exit at an angle that is not perfectly centered. If PID immediately reacts too strongly, Cheese can overcorrect and move toward a wall. Post-curve recovery gives the robot a short controlled transition after the curve.

```text
post_curve_active = True
post_curve_end_time = now + POST_CURVE_TIME
```

After the recovery time ends, post-curve mode turns off, PID resets, and the robot resumes normal driving. This makes curve exits smoother and helps prevent crashes right after turns.

---

## ❀ Behavior Priority Explanation ────୨ৎ────────୨ৎ────

Cheese’s flowchart works because behaviors are organized by priority. Without priorities, two parts of the program could try to control the robot at the same time.

| Priority | Behavior | Reason |
| :---: | :--- | :--- |
| **1** | Invalid sensor stop | The robot should not continue if important readings are unreliable. |
| **2** | Final parking logic | After the required laps, stopping correctly becomes the main objective. |
| **3** | Curve sequence handling | Curve events are used to track progress and control special movement states. |
| **4** | Post-curve recovery | The robot needs a smoother transition after each curve. |
| **5** | Wall protection | Prevents crashes when the robot gets too close to a wall. |
| **6** | PID centering | Handles normal straight movement when no higher-priority behavior is active. |

This priority structure prevents behavior conflicts. For example, final parking should not be interrupted by normal PID driving, and curve handling should not be counted twice because of repeated color sensor flicker.

---

## ❀ Engineering Importance of the Flowchart ────୨ৎ────────୨ৎ────

This flowchart is important because it connects the robot’s software to its physical behavior. Each block represents something Cheese must do on the mat: prepare its motors and sensors, align with the walls, drive between them, detect floor colors, count curves, recover after turns, and stop after the required laps.

The diagram also makes debugging easier. If Cheese fails during straight movement, the team can inspect ultrasonic readings, PID values, and wall protection. If it misses curves, the team can inspect color confirmation and curve pairing. If it stops too early or too late, the team can inspect lap counting and final stop references.

This makes the flowchart more than a picture. It becomes a map for testing. Each behavior block can be tested separately, which helps the team avoid changing unrelated parts of the code when only one subsystem is failing.

---

## ❀ Flowchart Summary ────୨ৎ────────୨ৎ────

The flowchart represents Cheese’s full Open Challenge navigation cycle. The robot begins by initializing its motors, sensors, constants, counters, and state variables. Before starting movement, it uses the ultrasonic sensors to check alignment so the run begins from a more stable position.

During the main loop, Cheese repeatedly reads the wall sensors and floor color sensor. If no confirmed color event is active, the robot uses PID centering and wall protection to stay between the walls. If a valid color is confirmed, the robot updates its curve sequence and counts a completed curve only when the opposite color completes the pair.

After enough curves are completed, the robot updates the lap counter. Once the required number of laps is reached, Cheese activates the final stop logic. The final stop is based on internal run references such as lap completion, timing, and movement position, not on a rear ultrasonic sensor.
