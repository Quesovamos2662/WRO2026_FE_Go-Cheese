# *3.1 Algorithm Architecture*

# ᯓ★ 3.1.1 Main Loop & Program Structure ᯓ★

This subsection covers how the program is organized and what it does on every cycle. It is the skeleton the rest of Section 3.1 hangs on: the other files develop the state, the steering logic, the PID, and the constants, but they all operate inside the structure described here.

---

## ❀ The Shape of the Program ────୨ৎ────────୨ৎ────

The program runs top to bottom in five blocks, each with one job. The overview (3.0) lists them; here is what each one contributes to how the robot runs:

| Block | Contribution |
| :--- | :--- |
| **Hardware setup** | Binds the two motors and three sensors to their ports and sets sensor modes: `US-DIST-CM` for the ultrasonics, `COL-COLOR` for the color sensor. Also resets the steering motor to a known zero. |
| **Settings** | Every tunable constant in one block (3.1.5). |
| **Program state** | The globals that carry information between cycles (3.1.2). |
| **Functions** | Single-responsibility functions grouped by job. |
| **Main loop** | The `while` loop that runs the robot until it finishes or is stopped. |

Keeping each function responsible for exactly one thing is what let us cut the program to roughly 300 lines without losing behavior. A short main loop is only possible when the functions it calls each do one job cleanly.

---

## ❀ What Happens Before the Loop ────୨ৎ────────୨ৎ────

Three things happen once, before the robot moves:

**1. The start sequence.** `show_ready()` displays "READY!" on the brick, and `wait_for_left_button()` waits for the left button. This function is written defensively: it first waits for the button to be released in case it was already held, then waits for a fresh press, then waits for release again. The result is that one physical press counts as exactly one start, with no accidental triggering.

**2. Starting-row detection.** `detect_starting_row()` averages five readings from both walls before the robot moves, and records which side reads closer as `STARTING_ROW`. This is used once, at the very end, so the robot parks on the same side it started. It is covered fully in Section 3.3. If either sensor cannot get a valid reading, `STARTING_ROW` stays `None` and the final alignment step is simply skipped, so a bad initial reading can never cause bad parking behavior.

**3. Zeroing the references.** The drive encoder is set to zero (`drive.position = 0`) and the start time is recorded. This gives the robot two clean references, distance and time, both beginning at zero the instant it starts moving. Everything the parking logic later computes is measured from these two zero points.

---

## ❀ The Main Loop ────୨ৎ────────୨ৎ────

Everything the robot does while running happens inside one `while` loop. Each cycle performs the same sequence in the same order:

**1. Check stop conditions first.** Before anything else, the loop checks whether it should end. The back button is a permanent emergency stop. If parking is active, the loop checks two independent stop conditions, a distance target and a time deadline, and ends if either is met. Checking this at the top of the cycle means the robot decides whether to stop *before* it decides how to move, so it never takes one more steering action after it should have already parked.

**2. Read the sensors.** `read_color()` returns the current line color or `None`; `read_walls()` returns both wall distances, already validated.

**3. Update the count.** `update_color_count()` advances the corner counter if a color pair has completed. This function only counts. It never steers.

**4. Decide how to steer.** The steering priority chain runs as an if/elif/else: wall protection first, then PID centering, then the single-sensor fallback (3.1.3).

**5. Sleep.** The loop pauses for `LOOP_DELAY`, which is 0.01 seconds, then repeats.

**The loop rate.** At a 10 ms delay, the robot completes roughly **100 cycles per second**. That means it re-reads its sensors and re-decides its steering about once every centimeter or two of travel at our speeds. This is fast enough that the PID sees the walls change smoothly rather than in jumps, which is what makes gentle, continuous correction possible instead of coarse reactions.

**The structural key of the loop** is that counting and steering are fully separate. They run in the same cycle but never touch each other: the color sensor updates the count, the ultrasonic sensors decide the steering, and neither reaches into the other's job. This separation is the single most important design decision in the whole program, and it is developed in 3.1.3.

---

## ❀ What Happens When the Loop Ends ────୨ৎ────────୨ৎ────

The loop exits for one of three reasons: the back button is pressed, the parking distance target is reached, or the parking time deadline passes.

When it exits because of parking, `align_to_starting_row()` runs once, nudging the robot onto its original starting side before it comes to rest. This step is bounded by a timeout so it can never hang or drift the robot into extra travel.

Whatever the reason for exiting, a **`finally` block** guarantees a clean shutdown: `stop_robot()` cuts power to both motors with braking engaged, and the screen shows "STOPPED". Because this lives in a `finally`, it runs even if something fails mid-loop. The robot cannot be left driving after the program has ended, which is a basic but important safety guarantee.

# ᯓ★ 3.1.2 State Variables ᯓ★

This subsection covers the variables that make up the robot's memory. Cheese has no map and no plan; what it has is a set of global variables that carry information from one loop cycle to the next. This is where the robot stores what it is doing, what it has done, and what it needs to remember to finish.

---

## ❀ Why State Matters Here ────୨ৎ────────୨ৎ────

Each loop cycle is nearly stateless on its own: it reads the sensors and decides how to steer from what it sees right now. But some things cannot be decided from a single instant. Whether a corner has been completed, how many laps have passed, where the robot needs to stop, all of these require remembering something from earlier. The state variables are that memory.

They fall into four groups: counting state, PID state, parking state, and the starting-row reference.

---

## ❀ Counting State ────୨ৎ────────୨ৎ────

These track the robot's progress through the corners and laps.

| Variable | Purpose |
| :--- | :--- |
| `curve_active` | True while the robot is between a first color and its matching exit color. This is a counting flag only; despite the name, it does not affect steering. |
| `required_exit_color` | The color that will close the current corner: the opposite of whichever color opened it. If blue opened the pair, this holds orange, and vice versa. |
| `last_accepted_color` | The last color that was counted. Prevents a single continuous line from being counted more than once while the sensor sits over it. |
| `candidate_color` | The color currently being considered for confirmation. |
| `candidate_count` | How many consecutive readings of `candidate_color` have been seen, compared against `COLOR_CONFIRM_READINGS` before a color is accepted. |
| `completed_curves` | The running count of completed corners. The run ends at 12. |
| `completed_laps` | Derived from `completed_curves`, one lap per four corners. |

---

## ❀ PID State ────୨ৎ────────୨ৎ────

The PID controller is not stateless. It needs to remember two things between cycles to do its job.

| Variable | Purpose |
| :--- | :--- |
| `integral` | The running sum of every error the controller has seen, clamped to ±70. This is what lets the PID respond to small, persistent drift that the proportional term alone is too weak to correct. |
| `last_error` | The error from the previous cycle. The derivative term is the difference between the current error and this, which is how the controller senses how fast the error is changing. |

Both are reset to zero by `reset_pid()` whenever wall protection takes over, because a correction that has been accumulating against a wall is no longer describing useful information once the robot breaks away from it.

---

## ❀ Parking State ────୨ৎ────────୨ৎ────

This is the most interesting group, because it holds the data the robot measures during its own run to figure out where to stop.

| Variable | Purpose |
| :--- | :--- |
| `start_time` | The moment the robot began moving. All timing is measured from here. |
| `first_color_delay` | How long after starting the robot saw its first color. Used as a time-based parking fallback. |
| `first_color_distance` | The encoder distance at the first color. Used as a distance-based parking fallback. |
| `lap1_position` | The drive-encoder position recorded when lap 1 completed. |
| `lap2_position` | The drive-encoder position recorded when lap 2 completed. |
| `stop_deadline` | The time at which the robot will stop, as a safety ceiling. |
| `stop_position_target` | The encoder distance at which the robot will stop, the primary parking condition. |
| `parking_active` | True once the final corner is counted and the robot is on its parking approach. |

The pairing of `lap1_position` and `lap2_position` is the clever part: because both are measured from the same physical reference point, the difference between them is the true length of one lap, measured from this run rather than assumed. Section 3.3 develops how the parking logic uses this.

---

## ❀ The Starting-Row Reference ────୨ৎ────────୨ৎ────

| Variable | Purpose |
| :--- | :--- |
| `STARTING_ROW` | Which wall the robot started closer to, `LEFT` or `RIGHT`, measured before it moves. |

This single variable is set once, before the run, and read once, at the very end, so the robot parks on the same side it began. It is deliberately isolated from everything else: it never touches the PID, wall protection, or counting. If it cannot be measured cleanly at the start, it stays `None` and the final alignment is simply skipped.

---

## ❀ A Note on Global State ────୨ৎ────────୨ৎ────

Every one of these lives as a global variable, modified by the functions that need it. For a program this size, that is a deliberate and reasonable choice: the state is small enough to hold in view all at once, and keeping it global avoids threading the same handful of values through every function call. The tradeoff is that a reader has to know which functions write to which variables, which is why each function that modifies state declares it explicitly with `global` at the top, making its effects visible at a glance.

# ᯓ★ 3.1.3 The Steering Priority Chain ᯓ★

This subsection covers how the robot decides which way to steer on every cycle. It is the most important logic in the program, because it is where the robot's behavior actually comes from. Everything else, reading sensors, counting corners, holds the robot together, but this is what drives it.

---

## ❀ The Chain ────୨ৎ────────୨ৎ────

Every cycle, after reading the sensors and updating the corner count, the robot runs a three-layer decision. It is a plain if/elif/else, and the first layer that applies takes the entire cycle:

| # | Layer | Condition | Function |
| :---: | :--- | :--- | :--- |
| **1** | **Wall protection** | Either wall inside its threshold | `wall_protection()` |
| **2** | **PID centering** | Both walls valid, no wall too close | `drive_pid()` |
| **3** | **Single-sensor fallback** | One or both readings invalid | `drive_safely_without_two_walls()` |

The order is the logic. Wall protection is checked first because avoiding a collision outranks everything. If it acts, it returns `True` and the cycle ends there. If no wall is close enough to trigger it, and both sensors are giving valid readings, the PID takes over and centers the robot. If the robot cannot see both walls, it falls through to the cautious fallback.

Only one layer ever runs per cycle. There is no blending, no averaging between layers. Each 10 ms tick, exactly one of these three is in control.

---

## ❀ Why the Color Sensor Is Not Here ────୨ৎ────────୨ৎ────

The most important thing about this chain is what is missing from it.

The color sensor is read on every cycle, right alongside the ultrasonics. But it is **not** part of the steering decision. `update_color_count()` runs before the chain, updates the corner count, and returns without ever touching the steering. Our own code comment states it directly: *color detection never changes steering.*

This means the robot corners the exact same way it drives a straight. There is no "corner mode" that seizes the wheel when a line appears. As the robot rounds a bend, the walls close and open around it, the ultrasonic readings change, and the PID and wall protection respond to those changing distances continuously. The corner is driven by the walls, not by the color. The color sensor is only counting how many corners have gone by.

We separated these two jobs on purpose. When we tried, in earlier versions, to make a sensor trigger and drive the turns directly, a single bad reading during a corner would send the robot into the wall (Section 2.3). Taking steering authority away from the corner-detection entirely removed that failure: a miscounted color now costs us a count, never a crash.

---

## ❀ Layer 1: Wall Protection ────୨ৎ────────୨ৎ────

`wall_protection()` is the highest-priority layer and it escalates in two stages.

| Stage | Trigger | Speed | Angle |
| :--- | :---: | :---: | :---: |
| **Severe** | wall ≤ 19.0 cm | −40 | 32° |
| **Gentle** | wall ≤ 21.5 cm | −46 | 18° |

The function checks the severe condition first. If either wall is at or within 19.0 cm, it drops to the severe speed and steers away sharply. Otherwise, if either wall is within 21.5 cm, it applies the gentler, slightly faster correction. `steer_away_from_wall()` picks the direction by steering away from whichever wall reads closer, and works correctly even if only one wall is visible.

Both stages call `reset_pid()`. This matters: while the robot has been drifting toward a wall, the PID's integral term has been accumulating error. Once wall protection pulls the robot away, that accumulated value is stale, it describes a situation that no longer exists. Clearing it prevents the PID from carrying that outdated correction into the next straight and overshooting the other way.

---

## ❀ Layer 3: The Single-Sensor Fallback ────୨ৎ────────୨ৎ────

`drive_safely_without_two_walls()` is what runs when the robot cannot see both walls. It is not clever, and it is not meant to be. It is a floor beneath the robot's behavior.

| Situation | Response |
| :--- | :--- |
| Only the left wall is visible | Steer away from it if it is too close, otherwise drive straight |
| Only the right wall is visible | Steer away from it if it is too close, otherwise drive straight |
| Neither wall is visible | Center the steering and drive straight at reduced speed |

The purpose is simple: when the robot loses its sensor data, it should keep moving carefully in a sane direction rather than doing something confident and wrong. It holds its heading until a sensor recovers and one of the higher layers takes back over.

---

## ❀ How Bad Readings Never Reach the Chain ────୨ৎ────────୨ৎ────

The chain trusts its inputs because the inputs are cleaned before they arrive. `read_walls()` passes every ultrasonic value through `valid_distance()`, which rejects anything outside 1.0 to 255.0 cm, physically impossible readings, zeros, and extreme spikes. A reading that fails becomes `None`, which the chain treats as "no wall visible" rather than as a real distance.

This is why a single spurious ultrasonic spike, the kind documented in Section 2.3, cannot by itself steer the robot into a wall. A bad reading does not produce a bad steering command; it produces no reading, and the fallback handles the absence safely.

# ᯓ★ 3.1.4 The PID Controller ᯓ★

This subsection covers the controller that keeps Cheese centered between the walls. It is the second layer of the priority chain, the one that runs whenever no wall is close enough to trigger protection, which means it is the layer the robot spends most of its time in. On a clean straight, the PID is the robot.

---

## ❀ What a PID Controller Does ────୨ৎ────────୨ৎ────

A PID controller is a way of correcting an error smoothly instead of in jerks. Rather than "too close to the left, turn right," it asks three questions at once: how far off am I, how long have I been off, and how fast is that changing. It combines the answers into a single, proportioned correction. The name comes from its three terms: Proportional, Integral, Derivative.

For Cheese, the error being corrected is how far off-center it is between the two walls, and the correction is a steering angle.

---

## ❀ The Error ────୨ৎ────────୨ৎ────

The controller computes one number every cycle:

`error = right − left`

This is the difference between the two wall distances. Reading it is straightforward:

| Reading | Meaning |
| :--- | :--- |
| **error > 0** | The right wall is farther away, so the robot has drifted left of center |
| **error < 0** | The left wall is farther away, so the robot has drifted right of center |
| **error = 0** | The robot is exactly centered |

The controller's entire job is driving that number toward zero.

---

## ❀ The Three Terms ────୨ৎ────────୨ৎ────

`correction = (KP × error) + (KI × integral) + (KD × derivative)`

Each term handles a different aspect of the error:

| Term | Value | What it responds to | Effect |
| :--- | :---: | :--- | :--- |
| **Proportional** | KP = 0.28 | The error right now | The bigger the drift, the bigger the correction |
| **Integral** | KI = 0.001 | Error accumulated over time | Corrects small, persistent drift the P term is too weak to catch |
| **Derivative** | KD = 0.08 | How fast the error is changing | Damps the response, preventing overshoot and oscillation |

The proportional term does most of the work. The integral term is deliberately tiny (0.001) because it accumulates every cycle and a larger value would build up too quickly on our noisy sensors. The derivative term is what stops the robot from overcorrecting and zig-zagging down the straight.

---

## ❀ From Correction to Steering Angle ────୨ৎ────────୨ৎ────

The raw correction is not sent to the motor directly. Two things happen to it first:

**It is negated.** The steering angle is `-correction`, because of how our steering direction maps to the error sign. This is a consequence of the physical mounting of the steering, captured in the `STEER_DIRECTION` constant elsewhere.

**It is clamped.** The angle is limited to ±24° (`MAX_PID_ANGLE`):

`steering_angle = clamp(-correction, -24, 24)`

This ceiling is important. It guarantees the PID can never command a turn sharp enough to be mistaken for a corner. No matter how large the error grows, the centering correction stays gentle, which keeps normal driving smooth and keeps the PID from ever throwing the robot into a hard turn on a straight.

---

## ❀ Integral Windup Protection ────୨ৎ────────୨ৎ────

The integral term has a known danger. Because it sums error every cycle, it can grow without limit if the robot is stuck in an error it cannot correct, for example pinned near a wall. Left unchecked, that runaway sum would eventually dominate the output and send the robot lurching once it finally broke free. This is called integral windup.

We prevent it two ways:

**A hard clamp.** The integral is limited to ±70 (`INTEGRAL_LIMIT`) every cycle, so it can never accumulate past a sane bound.

**A reset on wall protection.** Whenever `wall_protection()` takes over, it calls `reset_pid()`, zeroing the integral entirely. A robot fighting a wall has been accumulating error the whole time, and that error is meaningless once it pulls away, so it is cleared rather than carried forward.

---

## ❀ Why Our Gains Are So Gentle ────୨ৎ────────୨ৎ────

Our gains, KP 0.28 and KD 0.08, are far lower than the aggressive values we ran early on. Our own code marks this: *less-aggressive PID than the reference program.* The reason is mechanical, and it is one of the clearest cases on this robot of a software value being decided by a hardware fix.

Our early steering was stiff and imprecise (Section 1.3). A command to steer did not reliably produce the angle we asked for, so the software compensated by shouting: high gains, forcing a big response to guarantee any response at all. The cost was violent, twitchy corrections and a robot that oscillated.

When we freed the steering linkage and reinforced the assembly, the robot started producing the exact angles we commanded. The slack the software had been fighting was gone. So we turned the gains down, because a mechanism that responds accurately does not need to be forced.

> This is the lesson in one line: we did not tune our way out of the oscillation, we built our way out of it. The correct fix for a software problem turned out to be a hardware one, and the gains came down on their own once the steering was solid.
