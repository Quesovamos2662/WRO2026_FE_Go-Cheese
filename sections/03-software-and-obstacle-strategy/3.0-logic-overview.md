# ᯓ★ 3.0 Software Logic Overview ᯓ★

This section details Cheese's software architecture, covering the platform and libraries, a function-level breakdown of the code, the priority-based execution model, and the two control systems that drive the robot. The main navigation code is implemented in **Python 3 using the ev3dev2 framework, running directly on the EV3 brick.** A second processor, an Arduino Nano, handles the HuskyLens camera in C++, though vision is not yet integrated into the navigation loop.

Every section after this overview focuses on a single subsystem in full technical detail.

## ❀ 3.0.3 Design Principle: Minimum Sufficient Logic ────୨ৎ────────୨ৎ────

Our software went from over **1000 lines to just over 300**. This was not feature loss. Most of the deleted complexity had been compensating for the imprecise steering documented in Section 1.3, and once that was fixed in hardware, the software fighting it became dead weight.

What remains, in roughly 300 lines, still performs PID lane-centering, RGB color detection, symmetric curve execution, two-stage wall protection, corner and lap counting, time-based parking, sensor validation, and graceful failure handling. Simple does not mean it does little. It means every line earns its place.

---

## ❀ 3.0.4 Execution Model: One Loop, Five Priorities ────୨ৎ────────୨ৎ────

The program is a single loop. Each 10 ms cycle reads the color sensor, reads both ultrasonic sensors, then passes the result through a five-layer priority chain. The first layer whose condition is met takes control; every layer below is skipped that cycle.

| # | Layer | Condition | Core action |
| :---: | :--- | :--- | :--- |
| 1 | Active curve | curve in progress | Fixed-angle turn, uninterruptible |
| 2 | Post-curve recovery | within 0.16 s of curve end | Brief counter-steer |
| 3 | Wall protection | wall too close (two stages) | Steer away, slow down, reset integral |
| 4 | PID centering | both walls valid | Correction capped at a max angle |
| 5 | Single-sensor fallback | one or both readings invalid | Drive cautiously on remaining data |

## ❀ 3.0.5 Core Functions ────୨ৎ────────୨ৎ────

The program's work is divided across a small set of focused functions. This is the map of what each one does; the sections that follow develop the important ones in detail.

| Function | Role |
| :--- | :--- |
| `read_color()` | Reads the color sensor in RGB-RAW mode and returns which calibrated line color, if any, is currently seen. |
| `color_distance()` | Computes the Euclidean distance between a live RGB reading and a reference color, used to decide a match. |
| `get_distances()` | Reads both ultrasonic sensors and returns validated distances, discarding out-of-range values. |
| `valid_distance()` | Sanity-checks a single reading against the 1–255 cm physical range. |
| `pid_correction()` | Computes the PID steering correction from the left/right distance error. |
| `execute_curve()` | Drives a committed turn at a fixed angle until the exit color is seen. |
| `post_curve_correct()` | Applies the brief counter-steer that re-aligns the robot after a curve. |
| `check_walls()` | Runs the two-stage wall protection logic. |
| `navigate()` | The top-level decision function: evaluates the priority chain and dispatches to the winning layer. |
| `count_corner()` | Increments the corner counter and manages lap and parking state. |

*(Function names above should be matched to the exact identifiers in the source; adjust any that differ.)*

## ❀ 3.0.6 The Two Control Systems ────୨ৎ────────୨ৎ────

**Color-driven cornering.** Turns are triggered by the floor, not by distance. The color sensor runs in RGB-RAW mode; `color_distance()` measures how far a live reading sits from each calibrated line color, and a match within threshold triggers a turn. The logic is symmetric: whichever color appears first sets the direction, and the opposite color ends the turn. Developed in Section 3.5.

**PID lane-centering.** On straights, `pid_correction()` drives the error `right_distance − left_distance` toward zero.

| Gain | Value | Role |
| :--- | :---: | :--- |
| **KP** | 0.65 | Present error |
| **KI** | *[confirm]* | Accumulated error (clamped) |
| **KD** | *[confirm]* | Rate of change |

Developed in Section 3.1.

## ❀ 3.0.7 Reliability Layer ────୨ৎ────────୨ৎ────

Because our ultrasonic sensors produce erratic readings (Section 2.3), the software validates every reading against a physical range (1–255 cm) before use through `valid_distance()`. Any reading outside it is discarded. If one or both sensors fail validation, the robot degrades gracefully to single-sensor or straight-line driving rather than acting on bad data.
