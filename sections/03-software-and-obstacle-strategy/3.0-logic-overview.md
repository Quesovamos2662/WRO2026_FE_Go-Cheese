## ❀ Design Principle: Minimum Sufficient Logic ────୨ৎ────────୨ৎ────

Our software went from over **1000 lines to just over 300**. This was not feature loss. Most of the deleted complexity had been compensating for a mechanical problem, the imprecise steering documented in Section 1.3, and once that was fixed in hardware, the software fighting it became dead weight.

What remains is the minimum logic that still does everything the challenge requires. In roughly 300 lines, the program performs PID lane-centering, RGB color detection, symmetric curve execution, two-stage wall protection, corner and lap counting, time-based parking, sensor validation, and graceful failure handling. Simple does not mean it does little. It means every line earns its place.

---

## ❀ Execution Model: One Loop, Five Priorities ────୨ৎ────────୨ৎ────

The entire program is a single loop. Each 10 ms cycle reads the color sensor, reads both ultrasonic sensors, then passes the result through a **five-layer priority chain**. The first layer whose condition is met takes control; every layer below it is skipped that cycle.

| # | Layer | Condition | Core action |
| :---: | :--- | :--- | :--- |
| **1** | Active curve | `curve_active` set | Fixed 55° turn, uninterruptible |
| **2** | Post-curve recovery | Within 0.16 s of curve end | 16° counter-steer |
| **3** | Wall protection | Wall ≤ 21.5 cm (gentle) or ≤ 19.0 cm (severe) | Steer away, reduce speed, reset integral |
| **4** | PID centering | Both walls valid | Correction capped at 24° |
| **5** | Single-sensor fallback | One or both readings invalid | Drive cautiously on remaining data |

The ordering encodes urgency: a committed turn cannot be interrupted, an imminent wall outranks staying centered, and centering, the default state, sits at the bottom. Critically, an active curve **disables layers 3 and 4 entirely**, because ultrasonic readings are unreliable mid-corner (Section 2.3), so the color sensor becomes the only trusted input during a turn.
