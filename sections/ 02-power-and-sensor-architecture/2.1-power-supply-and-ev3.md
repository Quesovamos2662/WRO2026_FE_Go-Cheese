# бҜ“вҳ… 2.1 Power Supply & EV3 бҜ“вҳ…

Everything on Cheese runs off a single source: the EV3's rechargeable battery. There is no secondary power supply, no external pack, no separate feed for the camera. Every motor, every sensor, and the entire vision system draws from the same brick. This section explains how that power is distributed, and what happens when it starts running out.

---

## вқҖ 2.1.1 The Power Source в”Җв”Җв”Җв”ҖаӯЁа§Һв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҖаӯЁа§Һв”Җв”Җв”Җв”Җ

| Item | Detail |
| :--- | :--- |
| **Battery** | LEGO EV3 rechargeable battery pack |
| **Number of batteries** | 2, rotated during testing |
| **Operating voltage** | 9V (nominal) |
| **Powers** | EV3 brick, both motors, both ultrasonic sensors, color sensor, Arduino Nano, HuskyLens |

We use the official LEGO EV3 rechargeable battery pack, and we keep **two of them in rotation**. While one is on the robot, the other is charging. When the one in use runs down, we swap them and keep testing.

This was not an aesthetic choice. Testing sessions are long, and battery level turns out to affect how the robot behaves, so being able to swap in a fresh pack instead of waiting for a charge is what lets us keep running consistent tests instead of losing an afternoon to a charging cable.

---

## вқҖ 2.1.2 How Power Is Distributed в”Җв”Җв”Җв”ҖаӯЁа§Һв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҖаӯЁа§Һв”Җв”Җв”Җв”Җ

Cheese has a single power path, and every component sits somewhere along it:

| Component | Port | Powered by |
| :--- | :--- | :--- |
| **EV3 Large Motor** | `OUTPUT_B` | EV3 battery |
| **EV3 Medium Motor** | `OUTPUT_A` | EV3 battery |
| **Ultrasonic sensor (left)** | `INPUT_1` | EV3 battery |
| **Color sensor** | `INPUT_2` | EV3 battery |
| **Ultrasonic sensor (right)** | `INPUT_3` | EV3 battery |
| **Arduino Nano** | `USB` | EV3 battery, through the USB connection |
| **HuskyLens camera** | I2C to Nano | Arduino Nano |

The important detail is the last two rows. **The Arduino Nano is powered by the EV3 through the USB connection, and the HuskyLens is powered by the Nano.** The camera never touches the battery directly; it draws through the Nano, which draws through the EV3.

This means our entire vision system sits downstream of the same battery that drives the motors. Nothing on this robot has an independent supply.

---

## вқҖ 2.1.3 Why a Single Supply в”Җв”Җв”Җв”ҖаӯЁа§Һв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҖаӯЁа§Һв”Җв”Җв”Җв”Җ

Running the camera off the EV3 rather than giving it a separate battery was a deliberate choice, and it came with a real tradeoff.

**What it buys us.** One battery to charge, one battery to monitor, one thing that can run out. Adding a second power source would mean another component to keep charged, more wiring to route through an already crowded chassis, and more weight on a robot that is already at 888.1 g. It also means the camera powers up and down with the robot, with nothing to forget.

**What it costs us.** Everything shares a supply, so everything is affected when that supply weakens. There is no isolation: a battery that is draining is draining for the motors and the vision system at the same time. We accepted this because the alternative added complexity to a robot built by two people, and because in practice the failure we see from low battery is gradual and predictable rather than sudden.

## вқҖ 2.1.4 What Happens When the Battery Runs Low в”Җв”Җв”Җв”ҖаӯЁа§Һв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҖаӯЁа§Һв”Җв”Җв”Җв”Җ

This is the part we did not expect, and it is one of the most useful things testing has taught us.

**As the battery drains, the robot gets slower and it starts crashing.**

The slowdown makes intuitive sense: less voltage reaching the motors means less speed and less torque, so the same command produces less motion than it did on a full charge. But the crashing is the more interesting failure, and it took us a while to connect it to the battery at all.

The reason is that our steering is already fighting a timing problem. As documented in Section 1.3, the medium motor sometimes cannot reach the commanded steering angle before the robot has already entered the corner. Now consider what a weakening battery does to that: it slows the steering motor down. The motor that was already struggling to complete its swing in time is now completing it even slower, while the robot is still driving into the corner.

A low battery does not create a new problem. It takes the problem we already have and makes it worse.

**What this means for us.** Battery level is not a background detail on this robot; it is a variable that changes the robot's behavior. This is exactly why we rotate two packs during testing: a run on a half-drained battery is not testing the same robot as a run on a full one, and if we tuned our steering against a tired battery, we would be tuning against a moving target.

## вқҖ 2.1.5 Known Gaps в”Җв”Җв”Җв”ҖаӯЁа§Һв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҖаӯЁа§Һв”Җв”Җв”Җв”Җ

We want to be honest about what we have not done here.

We have **not measured the current draw** of each component, so we cannot yet present a proper power budget showing how much of the battery each subsystem consumes. We know from behavior that the motors dominate, but we do not have the numbers to prove it.

We have **not measured how long a charge lasts** under competition conditions, only that it lasts long enough for a testing session before performance visibly degrades.

These are the next things we would measure, because they would turn what we currently know by observation into something we know by number.
