# ᯓ★ 2.4 Sensor Calibration ᯓ★

Calibration on Cheese splits cleanly in two. The sensor that has to recognize something specific, the color sensor, was taught by hand on the real track. The ultrasonic sensors were never calibrated at all. This section explains both, defines what a threshold actually is and why ours exist, and is honest about the sensor we have not integrated yet.

| Sensor | Calibration method | Status |
| :--- | :--- | :--- |
| **Color sensor** | Taught on the real track lines, RGB thresholds derived from what it actually read | Calibrated and in use |
| **Ultrasonic (x2)** | None. Read raw. | Uncalibrated |
| **HuskyLens** | Will be trained on the real pillars | Not yet integrated into the code |

---

## ❀ 2.4.1 How the Color Sensor Identifies a Line ────୨ৎ────────୨ৎ────

Our color sensor runs in **RGB-RAW mode**. It does not hand us a color name. It emits its own light onto the floor, measures what bounces back, and returns three numbers: how much red, how much green and how much blue it is currently seeing.

We calibrated the sensor by reading the actual track lines and recording what came back. These are the reference values our code works from:

| Track line | R | G | B |
| :--- | :---: | :---: | :---: |
| **Orange** | 245 | 113 | 20 |
| **Blue** | 36 | 74 | 165 |

Read the two rows against each other and the reliability of our detection becomes obvious. Orange is overwhelmingly red-dominant with almost no blue. Blue is close to its inverse. These are not neighbouring colors a sensor might confuse. They sit near opposite ends of what it can see.

---

## ❀ 2.4.2 The Threshold: How We Decide What Counts as a Color ────୨ৎ────────୨ৎ────

The sensor never returns exactly (245, 113, 20) twice in a row. Ambient light shifts it, the sensor's height above the floor shifts it, wear on the mat shifts it. Every reading lands somewhere *near* the calibrated value rather than exactly on it.

This is why a color cannot be detected by equality. We cannot write "if the reading equals orange," because it almost never will.

**How our code solves it.** We treat each color as a point in three-dimensional RGB space, and we measure how far the current reading sits from each of our two reference points. The distance between two colors is calculated as:

`distance = sqrt( (R1 − R2)² + (G1 − G2)² + (B1 − B2)² )`

This is the Euclidean distance formula, the same one used to measure the gap between two points in physical space, applied to color instead. A reading that lands close to our stored orange produces a small distance to orange. A reading that lands nowhere near either color produces a large distance to both.

**The threshold is the acceptance radius.** Our code sets:

`COLOR_THRESHOLD = 45.0`

A reading is only accepted as orange or blue if its distance to that color is **45 or less**, and if it is closer to that color than to the other one. Both conditions must hold. If the reading is too far from both references, the code returns nothing at all, and the robot simply keeps driving.

Geometrically, the threshold defines a sphere of radius 45 around each of our two calibrated colors in RGB space. A reading inside the orange sphere is orange. A reading inside the blue sphere is blue. A reading outside both is neither, and it is ignored.

**Why the value matters.**

| Threshold too small | Threshold too large |
| :--- | :--- |
| The acceptance sphere shrinks. A legitimate reading of the orange line, shifted slightly by lighting, falls outside it and is rejected. | The sphere expands. Something that is not the line, a scuff, a shadow, a reflection, falls inside it and is accepted. |
| **Failure: false negative.** The robot drives past a turn it should have taken. | **Failure: false positive.** The robot turns where there is no turn. |

Calibration is the search for a radius wide enough to catch every real line and narrow enough to catch nothing else. Our 45 works because, as shown above, orange and blue sit so far apart in RGB space that even a generous radius around each one leaves an enormous gap between them.

---

## ❀ 2.4.3 Confirming a Reading ────୨ৎ────────୨ৎ────

Our code includes a mechanism for requiring a color to be seen repeatedly before acting on it:

`COLOR_CONFIRM_READINGS = 1`

This value defines how many consecutive matching readings the sensor must produce before the code treats the color as real. It is currently set to **1**, meaning a single valid reading is accepted immediately.

**Why we set it to 1.** Waiting for multiple consecutive readings introduces a delay, and at the speed the robot crosses a line, that delay costs us reaction time. Accepting the first valid reading means the second color triggers the post-curve correction without an extra loop of lag.

**What we give up.** A confirmation count of 1 means a single spurious reading, if one ever landed inside the threshold, would be acted on immediately. We accepted this because the threshold analysis above shows how unlikely a false positive is: for a reading to be wrongly accepted, it would have to land within 45 units of orange or blue in RGB space, and nothing else on our track is remotely close to either.

The structure is there if we need it. Raising this value to 2 or 3 is a one-line change if we ever start seeing false triggers.

---

## ❀ 2.4.4 Why We Taught It on the Real Track ────୨ৎ────────୨ৎ────

We did not take color values from a datasheet or from another team's code. We calibrated the sensor on the actual mat, reading the actual lines.

The reason is that **the orange on our track is not the orange in a specification.** The value the sensor reports is a product of the paint, the surface finish, the ambient lighting in the room, the sensor's own emitted light, and its exact height above the floor. Change any of those and the numbers change with them. A threshold borrowed from someone else's robot describes their conditions, not ours.

Calibrating on the real surface means our numbers, (245, 113, 20) and (36, 74, 165), describe the track we actually drive on, under the conditions we actually drive in.

**This is also why the color sensor is the only sensor that has never failed us.** There is no gap between what it was taught and what it sees.

---

## ❀ 2.4.5 The Ultrasonic Sensors Are Not Calibrated ────୨ৎ────────୨ৎ────

We will state this plainly rather than dress it up: **we do not calibrate our ultrasonic sensors, and we do not verify them before a run.** We read them raw.

**The reasoning we used.** After working with these sensors for a while, we could recognize a normal reading from a broken one by watching how the robot behaved. A formal pre-run check felt like ceremony rather than information.

**Why that reasoning has a hole in it.** We do get bad readings. Section 2.3 documents wild, uncontrolled distance values appearing after code changes and as the battery drains. Our process for handling them is entirely reactive: the robot misbehaves, we inspect the readings, we work out what the error probably was, and we change the code. We diagnose after the failure instead of catching it before.

That works, but every bad reading costs us an entire run. A startup check that compared both sensors against a known distance would take seconds and could catch a fault before it wastes a test.

We are naming this as a gap rather than defending it.

---

## ❀ 2.4.6 There Is No Startup Routine ────୨ৎ────────୨ৎ────

Cheese has no calibration phase at the beginning of a run. It does not pause, does not sweep its sensors, does not take reference readings. **It simply starts driving.**

Everything after that is calibration in motion. The robot reads both walls, works out where it sits between them, and continuously corrects toward the line it should be holding. The color sensor watches the floor for the orange that tells it a turn is coming. Cheese does not establish where it is before it moves. It works out where it is by moving.

**The tradeoff.** This is fast and simple, with no dead time before a run. But it means the robot has no baseline. It cannot compare a current reading against a known-good one, because it never took a known-good one. If a sensor comes up wrong, nothing in the system notices until the wrongness surfaces as a crash.

For a robot that already knows its ultrasonic readings are its least reliable input, that is the gap most worth closing.

---

## ❀ 2.4.7 The HuskyLens Is Not Calibrated Yet ────୨ৎ────────୨ৎ────

We want to be precise about the status of our camera rather than describe work we have not done.

**The HuskyLens is mounted on the robot, but it is not yet integrated into our code.** We are beginning that work now.

The calibration approach is decided: the HuskyLens learns by being shown, so we will train it on the **real competition pillars**, capturing each color and assigning it an ID that the camera stores and returns when it recognizes that color during a run. Our obstacle logic will act on that ID, sending the robot one way around a red pillar and the other way around a green one.

Training on the physical pillars rather than on a color swatch follows the same reasoning as the color sensor: a pillar under real lighting is not the same as a red on a screen, and the camera's saved reference needs to match the object it will actually meet on the field.

**Where lighting will matter.** This is the one place ambient light reaches our robot. Ultrasonic sensing is immune to it, since it works with sound. Our color sensor is nearly immune to it, since it supplies its own light. But the HuskyLens is a camera, and cameras see only what the light shows them. Room lighting affects how the pillars are illuminated, and that affects how confidently the camera identifies them. It is the one sensor input on Cheese that the venue itself can change.

---

## ❀ 2.4.8 Where We Stand ────୨ৎ────────୨ৎ────

| | Color sensor | Ultrasonic | HuskyLens |
| :--- | :---: | :---: | :---: |
| **Calibrated on real conditions** | ✅ | ❌ | Pending |
| **Verified before a run** | ❌ | ❌ | ❌ |
| **Affected by ambient light** | Barely | No | Yes |
| **Has ever failed us** | Never | Frequently | Not yet in use |

The pattern is not subtle. **The sensor we calibrated is the sensor that works.** The sensors we never calibrated are the ones that keep giving us trouble.

We are not claiming calibration is the whole explanation, because the cross-talk hypothesis in Section 2.3 and the battery voltage problem in Section 2.1 are both live candidates for our ultrasonic failures. But the correlation is there, and it is pointing somewhere.
