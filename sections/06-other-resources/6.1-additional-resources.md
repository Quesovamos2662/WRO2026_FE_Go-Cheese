## ❀ The Cheese Lab: Resource Vault ────୨ৎ────────୨ৎ────

This section works like a small **Cheese Lab** where all the extra resources are collected in one place. The main documentation explains the robot’s design, code, testing, tuning, and engineering decisions. This page adds the supporting material that makes the project easier to explore visually.

These resources help show that Cheese was not created from one final idea. It was built through testing, redesigning, recalibrating, documenting, and improving. Each graph, diagram, image, model, and calculation helps explain a different part of that process.

<p align="center">
  <strong>In simple words:</strong><br>
  This section is the evidence wall behind Cheese’s development.

## ❀ Graph Gallery Explanation ────୨ৎ────────୨ৎ────

<details>
<summary><strong>🧀 Cheese Development Recipe</strong></summary>

<br>

The Cheese Development Recipe graph summarizes the main “ingredients” used to improve Cheese. It connects mechanical behavior, sensors, software logic, lighting, parking, and documentation into one visual explanation.

This graph is useful because the robot did not improve from only one subsystem. The final performance depended on how different systems worked together.

<br>

**Related systems:** curves, color detection, wall protection, lighting, parking, and documentation.

</details>

<details>
<summary><strong>💻 Code Size Comparison</strong></summary>

<br>

The Code Size Comparison graph supports the decision to simplify the program. Earlier code versions were longer and harder to debug. Reducing the code structure made it easier to identify problems, tune specific values, and explain the software logic in the README.

<br>

**Related sections:** 3.1 Algorithm Architecture, 3.6 Tuning Process, and 4.1 Design Decision Log.

</details>

<details>
<summary><strong>⚖️ Robot Mass Tradeoff</strong></summary>

<br>

The Robot Mass Tradeoff graph compares Cheese v2 and Cheese v3. Cheese v3 became heavier, but the added structure supported more stable sensor placement, stronger mounting points, lighting, wiring, and the final competition layout.

<br>

**Related sections:** 1.1 Mechanical Design, 1.4 Chassis Explanation, and 5.1 Bill of Materials.

</details>

<details>
<summary><strong>🏎️ Speed Layer Chart</strong></summary>

<br>

The Speed Layer Chart explains why Cheese does not use the same speed for every situation. The robot moves faster during stable navigation, but slows down when wall protection becomes more important. This gives the steering system and sensors more time to react.

<br>

**Related systems:** PID driving, gentle wall correction, severe wall correction, steering response, and safety control.

</details>

<details>
<summary><strong>🔄 Curve Count Progress</strong></summary>

<br>

The Curve Count Progress graph supports the Open Challenge strategy. Since one lap has four curves, three laps require twelve confirmed curves. This graph helps connect curve counting to lap progress and final stop logic.

<br>

**Related sections:** 3.3 Open Challenge, 3.5 Corner Handling, and 3.6 Tuning Process.

</details>

<details>
<summary><strong>🛠️ Tuning Focus Map</strong></summary>

<br>

The Tuning Focus Map organizes the main systems that needed repeated calibration. It includes curve aggressiveness, turning angles, color detection, parking accuracy, obstacle detection, and lighting support.

<br>

**Related sections:** 3.6 Tuning Process and 4.2 What Did Not Work.

</details>

<details>
<summary><strong>🧪 Testing Journey</strong></summary>

<br>

The Testing Journey graph shows the repeated process used during development: identify an issue, connect it to a subsystem, adjust one value or method, test again, improve, and document the result.

<br>

**Related idea:** Cheese improved through testing cycles instead of random changes.

</details>
